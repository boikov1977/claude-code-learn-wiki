1. Зачем это нужно

Представьте: вы открыли холодильник, а там курица, три яйца и что-то подозрительное в банке. Вопрос — что с этим делать? Загуглить? Too many tabs open. Спросить у ChatGPT? Ну да, вы же каждый раз будете описывать свой холодильник текстом. А что, если просто сфотографировать и получить рецепт?

Cook — это AI-кулинарный помощник. Фотографируешь продукты телефоном, он распознаёт, что на фото, и генерирует рецепт. Три уровня сложности — от общажного повара до мишленовской звезды. Вот что мы получили в итоге:

2. Spec-Driven подход

Мы не писали код с порога. Сначала — спецификации. Инструмент — OpenSpec, встроенный в Claude Code через скиллы.

Цикл выглядел так:

flowchart LR A["proposal\nЧто делаем?"] --> B["design\nКак делаем?"] B --> C["specs\nДетали API"] C --> D["tasks\nЗадачи по файлам"] D --> E["code\nПишем код"] style A fill:#6c5ce7,stroke:#6c5ce7,color:#fff style B fill:#a29bfe,stroke:#a29bfe,color:#000 style C fill:#58a6ff,stroke:#58a6ff,color:#fff style D fill:#3fb950,stroke:#3fb950,color:#fff style E fill:#e1e4e8,stroke:#d0d7de,color:#000

Каждый шаг — отдельный файл. Всё лежало в /openspec/changes/ai-cook-assistant/ (сейчас заархивировано). Proposal описывал видение, design — архитектурные решения, specs — API-контракты и модели данных, tasks — конкретные файлы и что в них писать.

Почему spec-driven, а не «напишем потом документацию»?

Потому что когда субагент пишет код по спеке, у него нет опции «я тут додумаю сам». Спецификация — это контракт. Агент получает точное описание: какой файл, какая функция, какая сигнатура, какой ответ. Результат — код, который не нужно потом переписывать.

41 задача, разбитых на backend (7 модулей) и android (9 файлов). Каждая задача — конкретный файл с конкретным интерфейсом.

3. Прожарка архитектуры

Спецификации — это не Библия. Перед тем как начать писать код, мы провели прожарку — архитектурный ревью, который провёл прагматик-субагент.

«Давай-ка я посмотрю на ваши design-решения, пока вы не потратили 12 часов на код, который потом придётся переписывать.»
— Прагматик, наверное

Что нашли:

  • Дублирование моделей — три отдельных модели рецептов (DormRecipe, RestaurantRecipe, MichelinRecipe) вместо одной Recipe с опциональными полями. Прагматик объединил в одну.
  • CameraX в MVP — ~150 строк кода для камеры, когда система через Intent делает то же самое за 20 строк. Но об этом позже.
  • Навигация — NavHost с пятью экранами. Прагматик предложил sealed class ScreenState вместо навигации. Проще, менее подвержено багам, идеально для пяти экранов.
  • Rate limiting — без Semaphore для LLM-вызовов, OpenCode API мог бы вернуть 429. Добавили asyncio.Semaphore(1).

Итог прожарки: спецификации обновлены, код ещё не написан, а мы уже сэкономили себе часы работы. Это не паранойя — это инженерия.

LLMClient: Semaphore и fallback

OpenCode API — штука деликатная. Одновременно два запроса — и он может вернуть 429. Решение: asyncio.Semaphore(1). Один запрос за раз, остальные ждут:

llm_client.py
self._semaphore = asyncio.Semaphore(
    settings.max_concurrent_llm_requests  # = 1
)

async def chat_completion(self, model, messages, ...):
    async with self._semaphore:
        response = await self._client.post(
            "/chat/completions", json=payload
        )
        response.raise_for_status()
        return response.json()

Fallback работает просто: primary model → exception → fallback model. Если VISION_MODEL (mimo-v2.5) упал, пробуем qwen3.7-plus. Если оба упали — возвращаем пустой список. Пользователь видит «Продукты не распознаны», а не краш.

Парсинг JSON от LLM

LLM-модели иногда отвечают чистым JSON, иногда оборачивают в markdown code fence, иногда добавляют текст до/после. Утилита extract_json_object проходит три стадии:

services/utils.py
def extract_json_object(text: str) -> dict | None:
    # 1. Прямой парсинг
    result = json.loads(text)           # Чистый JSON?
    if isinstance(result, dict): return result

    # 2. Code fence
    fenced = extract_code_fence(text)   # ```json ... ```?
    result = json.loads(fenced)
    if isinstance(result, dict): return result

    # 3. Regex с подсчётом скобок
    # Ищем {...} с правильным балансом
    for i, ch in enumerate(text):
        if ch == "{":
            # ... балансируем depth ...

MIME spoofing и валидация

Кто-то попробует загрузить PNG с заголовком JPEG? Не пройдёт. Проверяем magic bytes:

routers/cook.py
# Валидация content-type
if image.content_type not in settings.allowed_image_formats:
    raise HTTPException(422, "Файл должен быть JPEG")

# Валидация magic bytes (анти-spoofing)
if contents[:3] != b"\xff\xd8\xff":
    raise HTTPException(422, "Файл не является JPEG")

# Валидация размера (bounded read, защита от OOM)
max_bytes = settings.max_image_size_mb * 1024 * 1024
contents = await image.read(max_bytes + 1)

5. Android: Jetpack Compose

Навигация без NavHost

Вместо навигационного графа — sealed class. Каждый экран — состояние. ViewModel хранит StateFlow<ScreenState>, Composable делает when:

CookViewModel.kt
sealed class ScreenState {
    data object Camera : ScreenState()
    data class AnalyzingProducts(...) : ScreenState()
    data class Products(val items: List<String>) : ScreenState()
    data class GeneratingRecipe(...) : ScreenState()
    data class Recipe(val recipe: RecipeData) : ScreenState()
    data class Error(val message: String) : ScreenState()
}

MainActivity.kt
@Composable
fun CookApp(viewModel: CookViewModel = viewModel()) {
    val state by viewModel.state.collectAsState()
    when (val s = state) {
        is ScreenState.Camera -> CameraScreen(...)
        is ScreenState.Products -> ProductScreen(...)
        is ScreenState.Recipe -> RecipeScreen(...)
        // ...
    }
}

Пять экранов, ни одной навигационной ошибки. backToCamera() просто ставит _state.value = ScreenState.Camera. Попробуй так сломать.

Network: Retrofit + Kotlin Serialization

Два метода в API-интерфейсе, один Retrofit-клиент с таймаутами и логированием:

CookApi.kt
interface CookApi {
    @Multipart
    @POST("api/cook")
    suspend fun recognizeProducts(
        @Part image: MultipartBody.Part,
    ): CookResponse

    @POST("api/recipe")
    suspend fun generateRecipe(
        @Body request: RecipeRequest,
    ): RecipeResponse
}

Logging включён только в debug. BuildConfig.DEBUG определяет уровень логирования — HEADERS в dev, NONE в release. Не засоряем Logcat на продакшене.

EXIF rotation

Смартфоны любят поворачивать JPEG по EXIF-ориентации. Без обработки фото от камеры приходит боком. Функция compressWithExifRotation читает ExifInterface, строит Matrix, применяет поворот + downscale, и сжимает обратно в JPEG:

CameraScreen.kt — compressWithExifRotation
val exif = ExifInterface(ByteArrayInputStream(jpegBytes))
val orientation = exif.getAttributeInt(
    ExifInterface.TAG_ORIENTATION,
    ExifInterface.ORIENTATION_NORMAL,
)

when (orientation) {
    ORIENTATION_ROTATE_90  -> matrix.postRotate(90f)
    ORIENTATION_ROTATE_180 -> matrix.postRotate(180f)
    ORIENTATION_ROTATE_270 -> matrix.postRotate(270f)
    // + flip, transpose, transverse
}

val result = Bitmap.createBitmap(
    bitmap, 0, 0, bitmap.width, bitmap.height,
    matrix, true
)

6. Большой пивот: CameraX → Intent

Это была самая болезненная и самая правильная вещь, которую мы сделали за всю разработку.

CameraX: что было

CameraX с ImageAnalysis, PreviewView, use cases, lifecycle-обвязка, permission handling, превью на экране, сохранение в MediaStore. 150+ строк кода. Три файла: CameraFragment, CameraConfig, CameraHelper.

И всё это для MVP. Для приложения, где пользователь делает одно фото и всё.

Intent: что стало

ActivityResultContracts.TakePicture() + FileProvider. Системная камера, минимум кода, zero permission (CAMERA не нужен, приложение не открывает камеру — оно просит систему это сделать).

CameraScreen.kt — вся логика камеры
val cameraLauncher = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.TakePicture(),
    onResult = { success ->
        if (success && photoFile.exists()) {
            val bytes = photoFile.readBytes()
            val compressed = compressWithExifRotation(bytes)
            onPhotoCaptured(compressed)
        }
    },
)

Button(onClick = { cameraLauncher.launch(photoUri) }) {
    Text("Сделать фото")
}

~20 строк кода вместо 150+. Не нужен CAMERA permission. Не нужно управлять lifecycle камеры. Не нужно делать превью. Системное приложение камеры делает всё это лучше, чем мы напишем за неделю.

Почему так правильно для MVP

  • MVP — это проверка гипотезы, а не production-ready приложение
  • Пользователь не различает, какая камера — системную или встроенную
  • Мы сэкономили ~130 строк кода и часы на отладку CameraX
  • Когда гипотеза подтвердится — переписать на CameraX будет тривиально

Удалили dependency на CameraX из build.gradle.kts, выпилили три файла, оставили один CameraScreen.kt. Чистота.

7. Code Review: 6 P1 findings

Код написан — не значит готово. Запустили code-review-workflow — самодельный workflow на TypeScript из .claude/workflows/.

flowchart LR Code["Код"] --> Reviewer["code-reviewer\n(DeepSeek v4-pro)"] Reviewer --> Report["6 P1 findings"] Report --> Fix["Исправления"] Fix --> Re-review["Re-review"] Re-review --> Pass["All clear"] style Reviewer fill:#cf222e,stroke:#cf222e,color:#fff style Report fill:#9a6700,stroke:#9a6700,color:#fff style Pass fill:#1a7f37,stroke:#1a7f37,color:#fff

Reviewer — read-only агент на deepseek-v4-pro. Он смотрит код и выдаёт findings. Все P1 (критические) нашли и пофиксили:

# Finding Что было Что стало
1 MIME spoofing Проверялся только content-type Добавлена проверка magic bytes \xff\xd8\xff
2 Rate limit Нет ограничения на LLM-вызовы Semaphore(1) в LLMClient
3 OOM на загрузке await image.read() без лимита read(max_bytes + 1) bounded read
4 EXIF rotation Фото могло прийти боком compressWithExifRotation с Matrix
5 LLM JSON парсинг Только прямой json.loads 3-уровневый парсинг: direct → fence → regex
6 Logging в debug HTTP logging всегда включён BuildConfig.DEBUG / DEBUG_LOGGING

Каждый finding — реальная проблема, которая могла бы сломать приложение на продакшене. Code review перед деплоем — не формальность, а необходимость.

9. Выводы и инсайты

Spec-driven работает

41 задача, ни одна не переписана. Когда спецификация чёткая, субагент пишет код правильно с первого раза. Не потому что модель умная, а потому что ей не нужно думать «а что имелось в виду».

Прожарка экономит время

Три модели рецептов вместо одной, NavHost вместо sealed class, CameraX вместо Intent — всё это нашли ДО написания кода. Один час ревью сэкономил десять часов переделки.

MVP = удалять, а не добавлять

Самое важное решение — выпилить CameraX. Не «давайте допилим», а «давайте уберём». 130 строк кода, которые не нужны для проверки гипотезы. MVP про минимальность, а не про «почти готово».

Многоагентный workflow

Пять разных агентов за одну сессию:

  • openspec-manager — спецификации
  • pragmatist — архитектурный ревью
  • android-coder + py-backend-coder — код
  • code-reviewer — ревью
  • tester-agent — тесты

Каждый делает своё дело. Никто не пытается быть всемогущим.

LLM-парсинг — это боль

LLM возвращают JSON по-разному: чистый, в code fence, с текстом вокруг. extract_json_object с тремя стадиями парсинга — это не перестраховка, а необходимость. Оставлять LLM додумывать — она не додумает.
