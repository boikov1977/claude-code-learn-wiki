Литературный отдел: дебаты архитекторов

Два раунда, три позиции, один вердикт — как спроектировали отдел из двух агентов

20 июня 2026 · 3 архитектора · 2 раунда · 7 файлов обмена через .triage/

1. Зачем нужен литературный отдел

У нас 12 AI-агентов, каждый со своей моделью, промптом и зоной ответственности. Вики растёт — уже 35 статей, и каждую нужно писать, редактировать, проверять на консистентность терминологии. Делать это вручную из основного Claude — значит постоянно переключать контекст между кодом и документацией.

Литературный отдел — это выделенная команда, которая занимается только контентом: статьи, README, CHANGELOG, docstrings. Отделяет документирование от разработки, даёт каждому агенту чёткую роль.

Ключевой вопрос дебатов: Достаточно ли двух агентов (writer + editor), или нужны ещё research-agent, consistency-checker и content-planner?

2. Исходный состав

На момент дебатов литературный отдел состоял из двух агентов:

Агент Модель Роль Режим
writer-agent claude-go-mimo-v2.5 Создание контента: статьи, README, CHANGELOG Write (редактирует файлы)
editor-agent claude-go-deepseek-v4-flash Редактура: стиль, грамматика, факты, терминология Read-only (только читает и комментирует)

    E -->|"severity: CRITICAL\nили WARNING"| W

    style W fill:#ddf4ff,stroke:#0969da,stroke-width:2px
    style E fill:#fbefff,stroke:#8250df,stroke-width:2px

Workflow простой: writer пишет, editor проверяет, writer фиксит. Максимум 2 цикла. Но вот что обсуждалось: хватает ли этого? Или нужны дополнительные агенты?

3. Вопрос к дебатам

Достаточно ли двух агентов, или нужно добавить:

  • research-agent — сбор материалов и первичных источников перед написанием
  • consistency-checker — глобальная проверка терминологии по всей вики
  • content-planner — планирование контента, roadmap статей

Запущены архитектурные дебаты (Triage) — 3 агента в 2 раунда.

Каждый архитектор выдвинул свою стратегию. Разброс оказался значительным — от 0 новых агентов до 3 сразу.

Суть: Добавить всех трёх, но запускать on-demand через Task Tool. Консистентность через RAG-индекс, а не отдельного агента.

Логика: Эфемерные агенты умирают после задачи — контекст не раздувается. RAG + knowledge graph решают то, что не может read-only editor. Масштабируется с 35 на 350 статей.

Стоимость: Разовая инвестиция в векторный индекс. DAG сложнее отлаживать, но research параллелится с write.

Суть: 0 новых агентов. Research — чек-лист в промпте writer. Consistency — bash-однострочник на grep.

Логика: Каждый новый агент = N токенов на каждый запуск. Для китайских моделей с ограниченным контекстом это критично. 35 статей — объём, который editor просматривает за один проход.

Решение Cost/мес Покрытие Риски
2 агента + скрипт ~0 доп. 90% Скрипт тупой, false positives
+ research-agent +5-10K токенов/статья 95% Дублирует writer, лишний round-trip
+ все 3 агента +15-30K токенов/статья 99% Раздутие контекста, медленнее, дороже
Только 2 агента без скрипта 0 75% Терминология расползается

Расширение: Только при появлении конкретных болей. Не раньше.

Innovator: от RAG к двухфазному режиму

Раунд 2

Принял от Pragmatist: grep-скрипт вместо RAG-агента для консистентности. Триггер-ориентированный подход к добавлению агентов.

Принял от Conservative: content-planner = человек. Стратегия не делегируется моделям.

Оставил за собой: Research ДО написания принципиально важен. Но не отдельный агент — а принудительный двухфазный режим внутри writer: research-note → draft. Не чек-лист, а явная структура.

Уточнил: consistency-скрипт вешать на PostToolUse хук writer-agent (после Write), не на pre-commit.

"Research ДО написания, а не ПОСЛЕ — это вопрос качества контента, а не удобства workflow."
— Innovator, round-2 rebuttal

Раунд 2

7. Консенсус и вердикт

После двух раундов все трое сошлись по ключевым пунктам:

CONSENSUS 2 агента — ядро (writer + editor). Никаких новых на старте.
CONSENSUS content-planner = человек (не модель). Стратегия контента не делегируется.
CONSENSUS Расширяться по триггерам, а не upfront. Не добавлять агентов «на вырост».
CONSENSUS consistency-скрипт на grep вместо отдельного агента. Бесплатно (0 токенов).
NOVELTY Research-фаза внутри writer — принудительный двухфазный режим (research-note → draft). Инициатива Innovator, принята всеми как бесплатное улучшение.

Принятые решения

# Решение Детали
1 2 агента сейчас writer-agent (mimo-v2.5) + editor-agent (deepseek-v4-flash). Без добавления новых.
2 consistency-скрипт Grep с фильтрацией контекста (исключать code blocks, цитаты). Вешается на PostToolUse хук writer.
3 Research-фаза Двухфазный режим в writer: research-note (poisk-mcp + search_web) → draft.
4 Триггеры расширения См. таблицу ниже.

Триггеры расширения

Агент Триггер Статус
research-agent (эфемерный) ≥2 CRITICAL от editor за месяц по фактическим ошибкам ОТЛОЖЕНО
consistency-checker (агент) 100+ статей + терминологический хаос ОТЛОЖЕНО
content-planner Никогда не делегировать модели ОТКЛЮЧЕНО

Что отложено

Предложение Кто предлагал Почему отложено
RAG + векторный индекс Innovator Overengineering для 35 статей. Вернуться при 100+
Эфемерный research-agent Innovator Заменён двухфазным режимом внутри writer
Полный отказ от consistency-скрипта Conservative Принят скрипт с фильтрацией как компромисс

8. Итоговый workflow

После дебатов литературный отдел работает по такой схеме:

flowchart TD
    START["writer-agent получает тему"] --> RNOTE["research-note\npoisk-mcp + search_web\nСбор 3-5 источников"]
    RNOTE --> DRAFT["draft\nЧерновик статьи"]
    DRAFT --> WRITE["Write: запись в файл"]
    WRITE --> HOOK["PostToolUse хук\nconsistency-скрипт (grep)\nфильтрация: code blocks, цитаты"]
    HOOK --> ED["editor-agent\nreview: стиль, факты, терминология"]
    ED --> SEV{"severity?"}
    SEV -->|"CRITICAL / WARNING"| FIX["writer-agent\nИсправления"]
    FIX --> RE["editor-agent\nПовторная проверка (макс 2 цикла)"]
    RE --> SEV
    SEV -->|"SUGGESTION или OK"| PUB["Публикация"]

    style START fill:#fff8c5,stroke:#9a6700,stroke-width:2px
    style RNOTE fill:#ddf4ff,stroke:#0969da,stroke-width:2px
    style DRAFT fill:#ddf4ff,stroke:#0969da,stroke-width:2px
    style WRITE fill:#f6f8fa,stroke:#1f2328,stroke-width:1px
    style HOOK fill:#dafbe1,stroke:#1a7f37,stroke-width:2px
    style ED fill:#fbefff,stroke:#8250df,stroke-width:2px
    style FIX fill:#ddf4ff,stroke:#0969da,stroke-width:1px
    style RE fill:#fbefff,stroke:#8250df,stroke-width:1px
    style PUB fill:#dafbe1,stroke:#1a7f37,stroke-width:3px

Ключевые особенности workflow:

  • Two-phase writer: research-note перед draft — принудительно, не по желанию
  • PostToolUse hook: consistency-скрипт запускается после каждого Write, ловит разночтения
  • Max 2 cycles: editor проверяет, writer фиксит, editor перепроверяет — не более 2 раз
  • Read-only editor: editor не пишет файлы, только комментирует
