Литературный отдел: как мы пишем и редактируем статьи

Полный обзор pipeline от черновика до публикации: 2 агента, 1 bash-скрипт, 9 шагов

20 июня 2026 · writer-agent + editor-agent · workflow с 2 циклами ревью

1. Что такое литературный отдел

Литературный отдел — это выделенная команда AI-агентов, которая занимается только контентом: статьи в вики, README, CHANGELOG, docstrings. Он отделён от основного разработческого workflow, чтобы документирование не переключало контекст основного агента.

Отдел состоит из двух агентов и одного bash-скрипта. Больше не нужно — это результат архитектурных дебатов трёх агентов-архитекторов (подробнее: Литературный отдел: дебаты архитекторов).

Принцип: Два агента + один bash-скрипт. Никаких новых субагентов, пока не сработает триггер (2+ CRITICAL по фактам за месяц, или 100+ статей в вики).

2. Состав отдела

✎
writer-agent
claude-go-mimo-v2.5
Агент-писатель: создаёт и обновляет контент

  • Bash Запускает git, consistency-check.sh, проверяет diff

📝
editor-agent
claude-go-deepseek-v4-flash
Senior technical editor: ревью и редактура (read-only)
  • Read-only Только читает и комментирует. Запрещены Write и Edit
  • poisk-mcp Проверка фактов и терминов через search_web
  • context7 Проверка актуальности API библиотек
  • Bash Проверяет файловую структуру, ссылки, git
🔍
consistency-check.sh
bash + grep
Проверка терминологии по всей вики (0 токенов)
  • grep -rPn Регулярные выражения по HTML-файлам вики
  • Filtering Исключает code blocks, цитаты и mermaid-диаграммы

Файлы конфигурации агентов хранятся в ~/.claude/agents/: writer-agent.md, editor-agent.md. Скрипт — scripts/consistency-check.sh.

3. Полный workflow: от темы до публикации

Вот как проходит создание каждой статьи — от получения темы до коммита в Git. Workflow включает 9 шагов с двумя циклами ревью от editor.

4. Каждый шаг подробно

1 Тема / запрос

Человек формулирует тему статьи. Это может быть конкретный запрос («Напиши про MCP») или результат дебатов архитекторов. Человек всегда остаётся стратегом — content-planner не делегируется модели.

2 Research-фаза

Writer-agent собирает материалы перед написанием. Это принудительная двухфазная модель (research-note → draft), а не опциональный чек-лист.

  • Поиск через search_web из poisk-mcp
  • Параллельный поиск: веерные запросы по аспектам темы
  • Чтение найденных источников
  • Верификация ключевых фактов
  • Запись research-note — структурированных заметок

Research-note содержит: найденные факты, источники, ключевые цитаты, структуру будущей статьи. Это промежуточный артефакт, который не публикуется.

3 Draft

На основе research-note writer-agent пишет черновик. Статьи всегда в формате HTML: полная страница с DOCTYPE, стилями, Mermaid-диаграммами и карточками.

  • Русский язык по умолчанию
  • Цветовая схема проекта (светлая тема, акцент #6c5ce7)
  • Mermaid-диаграммы: flowchart, sequence, classDiagram, stateDiagram, gantt
  • Футер с датой и ссылкой на вики

4 Запись в файл

Черновик записывается в wiki/articles/ через Write/Edit. Это единственный момент, когда writer-agent модифицирует файловую систему.

5 Consistency-check

Bash-скрипт scripts/consistency-check.sh проверяет терминологию по всей вики. Запускается writer-agent вручную после записи статьи, перед публикацией.

Что проверяет:

Паттерн Правильно Почему
сабагент субагент Единый термин в вики
Raунд (лат. R) Раунд Кириллическая Р, не латинская R
writer.agent writer-agent Дефис, не точка
permission.mode permissionMode camelCase в коде
context.7 context7 Без точки

Скрипт фильтрует совпадения в <code>, <pre> и mermaid-блоках — ловит ошибки только в прозе. При нарушениях возвращает exit code 1.

6 Editor review

Editor-agent проверяет статью по расширенному чек-листу. Результат — отчёт с findings, отсортированных по severity.

Чек-лист editor-agent

📃
Structure & Organization — иерархия заголовков (h1 → h2 → h3), логичная последовательность секций, нет «сиротских» подсекций
✎
Writing Style — грамматика, орфография, тон, лаконичность, активный залог, без канцелярита
🔍
Terminology — консистентность терминов (субагент/сабагент), расшифровка аббревиатур, корректные названия технологий
💻
Code Examples — форматирование, актуальность API, осмысленные комментарии, языковые теги
🔗
Cross-references & Links — небитые ссылки, консистентный формат, проверка файловой структуры
🎨
Visual & Formatting — выровненные таблицы, единообразные списки, alt-тексты у изображений
📖
Readability — абзацы по 3-5 предложений, сложные концепции разбиты на шаги, нет «стены текста»

Severity levels

Уровень Значение Блокирует публикацию? Примеры
CRITICAL Фактическая ошибка, битая ссылка, латинская R в слове Да Не тот API, сломанный href, Raунд вместо Раунд
WARNING Терминология, капитализация, формат, структура Требует исправления, но не блокирует Сабагент, неправильный регистр model ID, нумерация
SUGGESTION Предложение по улучшению Нет Добавить диаграмму, сократить абзац, уточнить формулировку

Отчёт сохраняется в файл _editor_review_<target>_<cycle>.md рядом со статьёй. Writer-agent читает отчёт и исправляет все CRITICAL и WARNING.

Workflow уже отработан вживую на статье «Литературный отдел: дебаты архитекторов». Вот как прошли два цикла ревью.

Раунд 1: первая проверка

Severity Что нашли Как исправили
CRITICAL Битые ссылки на .triage/ файлы Проверены пути, ссылки исправлены
CRITICAL Raунд — латинская R вместо кириллической Заменено на Раунд
WARNING Модели без префикса claude-go- Добавлены полные ID моделей

Итого Раунд 1: 2 CRITICAL, 2 WARNING, 1 SUGGESTION. Все CRITICAL и WARNING исправлены writer-agent.

Раунд 2: повторная проверка

Severity Что нашли Как исправили

✅
Результат: 0 CRITICAL, 0 WARNING после Раунда 2. Статья опубликована. Общее время: research + draft + 2 цикла ревью = 1 сессия.

6. Архитектура взаимодействия

Агенты не общаются напрямую. Взаимодействие идёт через файлы: writer записывает статью, editor читает и пишет отчёт, writer читает отчёт и правит. Это «файловый обмен», а не диалог. 

sequenceDiagram
    actor User as Человек
    participant W as writer-agent
    participant F as Файловая система
    participant CS as consistency-check.sh
    participant E as editor-agent

    User->>W: Тема статьи
    W->>F: research-note (промежуточный)
    W->>F: wiki/articles/target.html
    F->>CS: consistency-check.sh
    CS-->>W: Exit code + violations
    W->>E: Файл target.html
    E->>F: _editor_review_target_1.md
    F-->>W: Отчёт ревью
    W->>F: Исправленный target.html
    W->>E: Файл target.html (v2)
    E->>F: _editor_review_target_2.md
    F-->>W: Финальный отчёт
    W->>F: git commit + push
    W-->>User: Статья опубликована

Ключевые особенности архитектуры:

  • Нет прямой связи между writer и editor — только через файлы
  • Read-only editor не может случайно переписать текст
  • Ручной запуск consistency-check перед публикацией
  • Файлы отчётов сохраняются рядом со статьёй для трассировки

7. Почему именно 2 агента (а не больше)

В ходе архитектурных дебатов рассматривались три варианта: добавить 0, 1 или 3 новых агента. Вердикт: 2 агента + bash-скрипт оптимальны для текущего масштаба.

Альтернатива Стоимость Покрытие Почему нет
Только writer без editor 0 ~75% Терминология расползается, ошибки не ловятся
2 агента + скрипт (текущий) ~0 доп. ~90% —
+ research-agent +5-10K токенов/статья ~95% Дублирует writer, лишний проход
+ все 3 агента +15-30K токенов/статья ~99% Раздутие контекста, медленнее, дороже

Триггеры расширения

Новые агенты добавляются только при конкретных болях:

Агент Триггер Формат Статус
research-agent ≥2 CRITICAL по фактам за месяц Эфемерный (через Task Tool) Ожидание
consistency-checker 100+ статей + терминологический хаос Постоянный агент Ожидание
content-planner Никогда — Отключено
⚠
content-planner = всегда человек. Стратегия контента не делегируется слабым моделям. Это единственное «никогда» в системе.

8. Советы для тех, кто строит свой editorial pipeline

✔
Начни с 2 агентов. Writer (пишет) + editor (только читает). Read-only для editor — принципиально. Это предотвращает случайную перезапись текста.
✔
Файловый обмен вместо диалога. Агенты не общаются напрямую — пишут файлы (статьи, отчёты). Это проще дебажить и трассировать.
✔
Severity levels. CRITICAL / WARNING / SUGGESTION — простая классификация, которая делает автоматизацию возможной. CRITICAL = блокирует, WARNING = требует, SUGGESTION = опционально.
✔
Max 2 цикла. Не давай агентам зациклиться. Два прохода editor — достаточно для 90% статей. Остальное — SUGGESTION на потом.
✔
Consistency на bash, не на LLM. Grep по HTML-файлам бесплатный (0 токенов) и мгновенный. LLM-агент для проверки терминологии — overengineering до 100+ статей.
✔
Research перед написанием. Принудительная двухфазная модель (research-note → draft) повышает качество контента. Не чек-лист, а явный шаг.
✔
Расширяй по триггерам. Не добавляй агентов «на вырост». Определи конкретные триггеры (2+ CRITICAL/мес, 100+ статей) и расширяйся только когда они сработают.
