Строим команду AI-агентов для разработки

От идеи до 11 субагентов: coding-агенты, архитекторы-дебатёры, специалисты. Полный разбор архитектуры, моделей, промптов и workflow-паттернов.

🎯 Зачем строить команду агентов

Классический Claude Code — это один агент, который делает всё. Он пишет код, ревьюит, тестирует, документирует. Но есть нюанс: один агент, который делает всё, делает всё одинаково. Он не может одновременно быть скептическим ревьюером и смелым инноватором.

Идея простая: разбить разработку на роли. Каждая роль — это отдельный субагент со своей моделью, своими инструментами, своими ограничениями и своим промптом. Родительская сессия Claude Code выступает оркестратором, который решает, кого вызвать и когда.

Ключевой инсайт: Субагенты в Claude Code — это не просто разные промпты. Это полностью изолированные сессии со своими MCP-серверами, скиллами и наборами инструментов. Родительская сессия их не видит.

code-reviewer: read-only ревьюер

Самый строгий агент в команде. У него нет доступа к Write/Edit — он не может случайно «помочь» кодом. Только читает и выносит вердикт.

Важно: code-reviewer использует deepseek-v4-pro (более мощную модель), потому что ревью требует глубокого понимания контекста, а не скорости генерации кода.

Система severity в ревью:

  • BLOCKER (красный) — критическая проблема, блокирует деплой
  • CRITICAL (оранжевый) — серьёзная проблема, требует исправления
  • MAJOR (жёлтый) — значимая проблема, стоит исправить
  • NIT (серый) — мелочь, рекомендация

🏛️ Архитекторы: 3 агента для дебатов

Самый нестандартный элемент системы. Три архитектора не просто предлагают решения — они спорят друг с другом через файлы в .triage/.

🚀

innovator

model: claude-go-minimax-m3

Смелые решения. Новые технологии. Смотрит на 2-3 шага вперёд. Не боится ломать устаревшее.

⚖️

pragmatist

model: claude-go-deepseek-v4-pro

Баланс. Cost-benefit анализ. Инкрементальные улучшения. Таблицы сравнения — его конёк.

🛡️

conservative

model: claude-go-qwen3.7-plus

Стабильность. Проверенные решения. Указывает на риски и подводные камни новых подходов.

Все общение идёт через файлы: .triage/<task-name>/round-1/ и .triage/<task-name>/round-2/. Это обеспечивает:

  • Изоляцию — каждый агент не видит других во время раунда 1
  • Аудит — каждое предложение записано в файл
  • Ограничение длины — раунд 1: до 300 слов, раунд 2: до 200 слов
Зачем разные модели? Каждый архитектор использует свою модель, чтобы гарантировать разнообразие взглядов. Minimax-m3 генерирует смелые идеи, deepseek-v4-pro балансирует, qwen3.7-plus консерватична. Это не баг — это фича.

⚙️ Технические решения, которые стоят за системой

1. Изоляция агентов через frontmatter

Каждый агент — это Markdown-файл в ~/.claude/agents/ с YAML-frontmatter. MCP-серверы, скиллы и инструменты объявляются в frontmatter, и родительская сессия их не видит.

--- name: py-backend-coder description: > Use this agent when you need to write, refactor, or debug Python and backend code (FastAPI, SQLAlchemy, Celery). tools: - Read - Write - Edit - Bash - Glob - Grep model: claude-go-deepseek-v4-flash mcpServers: - name: context7 type: remote url: https://mcp.context7.com/mcp headers: CONTEXT7_API_KEY: ctx7sk-... skills: - fewer-permission-prompts permissionMode: auto ---

Ключевые поля:

ПолеНазначениеПример
model Какую модель использовать claude-go-deepseek-v4-flash
mcpServers MCP-серверы, доступные только этому агенту context7 для документации API
tools Разрешённые инструменты Read, Write, Edit, Bash, Glob, Grep
disallowedTools Запрещённые инструменты (для read-only) Write, Edit (у code-reviewer)
skills Скиллы, доступные агенту fewer-permission-prompts
permissionMode Режим прав auto (автоматическое подтверждение)

2. context7: единый источник правды для API

Все coding-агенты и архитекторы используют context7 для проверки актуальных API библиотек. Это критично, потому что китайские модели могут «галлюцинировать» API, которого нет.

# FastAPI: dependency injection npx ctx7 docs fastapi "dependency injection with async SQLAlchemy session" # SQLAlchemy 2.0: async patterns npx ctx7 docs sqlalchemy "selectinload vs joinedload async" # Pydantic v2: computed fields npx ctx7 docs pydantic "v2 computed fields serialization" # React: hooks npx ctx7 docs react "useTransition with Suspense" # Playwright: patterns npx ctx7 docs playwright "page.waitForResponse pattern"
Проблема, которую решает context7: Модели deepseek/mimo/qwen иногда генерируют API из Flask 1.x для FastAPI-задач, или используют Pydantic v1 syntax для v2. Context7 даёт актуальную документацию из первых рук.

3. Выбор моделей

Все модели китайские, работают через OpenCode-прокси. Прокси просто передаёт model ID в OpenCode Go — сам не решает, какую модель использовать.

МодельРольПочему именно она
deepseek-v4-flash Кодинг (python, frontend, backend) Быстрая, хорошая в генерации кода, дешёвая для частых вызовов
deepseek-v4-pro Code review + pragmatist Более мощная, лучше анализирует и находит проблемы
mimo-v2.5 Тесты + документация Хороша в структурированном выводе и detail-oriented задачах
minimax-m3 Инноватор Креативная, генерирует нестандартные решения
qwen3.7-plus Консерватор Осторожная, акцентирует на рисках
inherit explorer, openspec, statusline Наследует базовую модель (deepseek-v4-flash) — достаточно для простых задач

4. permissionMode: auto

Все coding-агенты используют permissionMode: auto. Это значит, что агент может свободно читать и писать файлы, не спрашивая разрешения у пользователя. Без этого каждый вызов субагента превращался бы в цепочку промптов «Allow Read? Allow Write? Allow Bash?».

5. fewer-permission-prompts

Скилл, который уменьшает количество запросов на подтверждение. Используется на всех coding-агентах в дополнение к permissionMode: auto.

🛠️ How-to: построй свою команду

Пошаговый рецепт для разработчика, который хочет повторить наш подход.

Шаг 1: Определи роли

Не создавай агентов «на всякий случай». Начни с 2-3 и расти по мере потребности. Минимальный набор:

Must-have (старт)

  • Coder — пишет код (выбери стек: Python, JS, или оба)
  • Reviewer — ревьюит код (read-only, строгая модель)

Nice-to-have (рост)

  • Tester — пишет и запускает тесты
  • Explorer — быстрый поиск по коду
  • Writer — документация

Advanced (команда)

  • Architect cluster (2-3 архитектора для дебатов)
  • Spec manager (spec-driven workflow)
  • Frontend/Backend split (когда проекты становятся большими)

Шаг 2: Выбери модели

Принцип: кодинг — быстрая модель, анализ — мощная модель, специалисты — inherit.

Совет: Если нет бюджета на разные модели — все на inherit. Качество промпта важнее модели. Хороший промпт на дешёвой модели лучше плохого промпта на дорогой.

Шаг 3: Напиши промпт

Структура промпта для coding-агента (после frontmatter):

# Название роли — Должность Ты — [роль]. Твоя задача — [что делаешь]. ## Контекст - Ты не додумываешь API. Если не знаешь — используй context7. - Ты пишешь код, а не рассуждаешь. - Ты даёшь код в работу. ## Когда тебя вызывают 1. Прочитай структуру проекта 2. Определи стек и зависимости 3. Реализуй с чек-листом качества 4. Напиши тесты 5. Проверь что всё проходит ## Чек-лист качества - [конкретные пункты для стека] ## Рабочий процесс ### Фаза 1: Анализ ### Фаза 2: Реализация ### Фаза 3: Контроль качества ## Формат ответа **Сделано:** ... **Файлы:** ... **Тесты:** ...

Шаг 4: Настрой изоляцию

  • Каждый агент — отдельный файл в ~/.claude/agents/
  • MCP-серверы — в frontmatter, не в ~/.claude.json (иначе все агенты увидят)
  • permissionMode: auto на всех coding-агентах
  • Read-only агенты: disallowedTools: [Write, Edit]

Шаг 5: Протестируй

Вызови каждого агента на реальной задаче и проверь:

# В Claude Code: Agent(name="py-backend-coder", prompt="Напиши REST API для управления задачами") Agent(name="code-reviewer", prompt="Проведи ревью изменений в git diff") Agent(name="tester-agent", prompt="Напиши тесты для src/tasks.py") Agent(name="explorer", prompt="Найди все файлы, которые используют SQLAlchemy")

⚠️ Ограничения и антипаттерны

Что не работает

АнтипаттернПочему плохоЧто делать
Дублирующие MCP в родителе и агенте Конфликты scope, лишние промпты подтверждения MCP только в frontmatter агента
10+ coding-агентов Сложно поддерживать, дублирование промптов 3 coding + 1 reviewer + 1 tester — оптимум
Все агенты на inherit Нет диверсификации взглядов Хотя бы reviewer на другой модели
Агент без чек-листа Качество зависит от «настроения» модели Конкретные пункты проверки в каждом агенте
Один агент и пишет, и ревьюит Конфликт интересов Отдельный reviewer с disallowedTools

Что работает хорошо

  • context7 на каждом coding-агенте — убирает 80% галлюцинаций API
  • permissionMode: auto — агенты не застревают на промптах
  • Формат ответа в конце промпта — структурированный output
  • Разные модели для архитекторов — настоящая диверсификация мнений
  • read-only reviewer — не может «помочь» кодом
  • Review перед коммитом — обязательный гейт качества
  • 3-фазовый workflow — баланс между структурой и гибкостью
