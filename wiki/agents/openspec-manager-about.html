OpenSpec Manager

Агент для spec-driven разработки: управление изменениями от proposal до архивации

model: inherit permissionMode: auto 11 skills изолированный MCP

1. Кто такой OpenSpec Manager

OpenSpec Manager — это специализированный субагент, который управляет spec-driven development (SDD) ворклоу по методологии OpenSpec. Он берёт на себя всю организационную работу: от создания proposal и проектирования до реализации задач и финальной архивации завершённых изменений.

Когда вызывать

  • Начало новой фичи — нужен proposal, дизайн и план задач
  • Рефакторинг или изменение архитектуры — требуется структурированный подход
  • Любая работа, где нужны спецификации и план перед написанием кода
  • Когда есть активное изменение — агент предложит продолжить

Роль в экосистеме

Родительский агент (пользовательский Claude Code) делегирует OpenSpec Manager-у управление изменениями. Менеджер не начинает реализацию без явной команды, спрашивает уточнения при неясностях, и после каждого этапа возвращает сводку: что сделано, какие артефакты созданы, сколько задач выполнено.

2. Инструменты

Агент имеет доступ к шести базовым инструментам, которые позволяют читать, писать, редактировать и исполнять команды в проекте.

3. Изолированный MCP: openspec-mcp

OpenSpec Manager имеет собственный MCP-сервер openspec-mcp, который запускается через uvx openspec-mcp. Этот сервер объявлен непосредственно в конфигурации агента, а не в глобальном settings.json, что делает его изолированным — доступным только этому агенту.

Чтение и навигация

  • list_changes — список активных изменений
  • list_specs — список спек по доменам
  • show_change — детальная информация об изменении
  • read_spec — чтение spec-файла по пути
  • read_tasks — чтение задач из tasks.md

Действия и валидация

  • validate_change — проверка корректности структуры изменения
  • archive_change — быстрая архивация завершённого изменения

MCP-сервер удобен для быстрых read-only запросов. Для полноценного workflow (создание proposal, реализация, синхронизация) используются скиллы.

5. Структура проекта OpenSpec

Корень OpenSpec-проекта — директория openspec/ в репозитории. Она содержит три ключевые зоны: спек-файлы (source of truth), активные изменения и архив завершённых.

specs/ — основные спецификации

Source of truth по доменам. Каждый домен — директория с файлом spec.md. Сюда вливаются дельта-спеки после синхронизации.

changes/ — активные и завершённые

Активные изменения лежат в changes/<name>/. Каждое содержит proposal, design, tasks.md и папку specs/ с дельта-спеками. Завершённые перемещаются в changes/archive/ с префиксом даты.

6. Workflow: полный жизненный цикл изменения

Работа над любым изменением проходит через четыре обязательных этапа. Каждый этап имеет свои артефакты и использует определённые скиллы.

Этап 1: Propose

Родительский агент даёт команду «Начни работу над X». OpenSpec Manager читает контекст кода, затем через скилл openspec-propose создаёт полный набор артефактов в openspec/changes/<name>/: proposal (что и почему), design (как делаем), tasks.md (чеклист задач) и дельта-спеки.

Этап 2: Apply

После утверждения proposal родительский агент говорит «Реализуй изменение X». Менеджер читает tasks.md, затем последовательно выполняет задачи через openspec-apply-change. Если задача неясна — не гадает, а запрашивает уточнение. После каждой задачи отмечает прогресс.

Этап 3: Sync

После реализации всех задач дельта-спеки из changes/<name>/specs/ синхронизируются с основными specs/ через openspec-sync-specs.

Этап 4: Archive

Финальный этап — архивация. Изменение перемещается в changes/archive/YYYY-MM-DD-<name>/ через openspec-archive-change. Родительский агент получает сводку.

8. Экосистема агента

OpenSpec Manager — центральный элемент, связывающий инструменты, скиллы, изолированный MCP и файловую систему проекта.

Режимы работы

model: inherit

Агент использует ту же модель, что и родительский агент. Не требуется отдельного API-ключа или дополнительных затрат на подписку.

permissionMode: auto

Все разрешённые команды выполняются автоматически, без запроса подтверждения у пользователя. Ускоряет работу при частых вызовах Bash и File Operations.

Три способа взаимодействия с OpenSpec

Способ Инструмент Когда использовать
Скиллы Skill tool Полный workflow: propose, apply, sync, archive. Содержит все шаги и проверки.
MCP-сервер Inlined MCP (uvx) Быстрые read-only запросы: список изменений, чтение задач, валидация.
CLI напрямую Bash Кастомные операции: openspec list --json, openspec status.
