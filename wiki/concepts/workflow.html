🏭 Что такое Workflow?

Представь конвейер на заводе. Сырьё загружается с одного конца — проходит через станции, на каждой делается своя операция — а с другого конца выходит готовое изделие. Никто не бегает между станциями и не передаёт вручную — всё автоматизировано.

Workflow — это скрипт, который описывает: каких агентов запустить, в каком порядке, что каждому дать на вход и что делать с результатами. Как конвейер: на одном конце задача, на другом — результат, собранный из работы нескольких агентов.

Проще простого: Workflow — это рецепт. Ты говоришь: «возьми задачу, отдай агенту A, его результат — агенту B, параллельно запусти агентов C и D, собери всё в финальный ответ». И Claude Code сам это исполняет — без твоего участия на каждом шагу.

Workflow описывается на JavaScript (не TypeScript) и живёт в файле .workflow.js. Это не просто «запусти агента» — это полноценный сценарий с фазами, параллельными блоками и сборкой результатов.

🔍 Отличие от обычного запуска агента

Обычный запуск

Ты вызываешь одного агента. Он получает задачу, делает её и возвращает результат.

Я: «innovator, придумай архитектуру» innovator: «Вот моя архитектура: ...» # Готово. Один агент, один ответ.

Workflow

Ты запускаешь сценарий. Он сам решает кого когда запускать, синхронизирует результаты, собирает финальный ответ.

Я: «запусти workflow дебатов» # Workflow сам: # 1. Запускает 3 агентов параллельно # 2. Ждёт все ответы # 3. Запускает их снова с чужими ответами # 4. Собирает финальный вердикт # Готово. Ноль ручных шагов.
Аспект Обычный запуск Workflow
Кто оркестрирует Ты (ручками) Скрипт (автоматически)
Параллелизм Ты сам решаешь, что параллельно Декларативно: parallel()
Передача данных Копируешь ответ одного — в промпт другого Автоматически: результат → вход следующего
Повторяемость Каждый раз руками Один скрипт — бесконечные запуски
Когда удобно 1–2 агента, простая задача 3+ агента, сложная оркестрация

Есть несколько проверенных схем организации мульти-агентных workflow. Выбор паттерна зависит от задачи.

Pipeline — конвейер

Суть: A → B → C. Каждый следующий агент получает результат предыдущего.

Пример: Исследователь находит баг → аналитик оценивает воздействие → фиксер пишет патч → ревьюер проверяет.

Когда: Когда каждый шаг зависит от предыдущего. Линейная обработка.

Fan-out — веер

Суть: Один запрос → N агентов параллельно → собираем ответы.

Пример: Код-ревью по 5 измерениям: безопасность, производительность, читаемость, тестопригодность, архитектура. Пять агентов работают одновременно.

Когда: Когда задача раскладывается на независимые подзадачи.

Adversarial — дебаты

Суть: Один агент даёт ответ, второй пытается опровергнуть, третий судит.

Пример: Наша схема дебатов — innovator, pragmatist, conservative спорят, writer-agent собирает вердикт.

Когда: Когда нужен сбалансированный взгляд. Хорошо для архитектурных решений.

Loop-until-dry — цикл до победы

Суть: Запускать агента, пока он находит баги или проблемы. Когда их нет — стоп.

Пример: Агент-тестер находит баг → фикс → снова тестер → снова баг → снова фикс... пока тестер не скажет «всё чисто».

Когда: Итеративные процессы: ревью, тестирование, оптимизация.

💡 Можно комбинировать: Fan-out для ревью → Adversarial для спора по найденным проблемам → Loop для итеративного исправления. Один workflow может содержать несколько паттернов.

💻 Как это выглядит в коде

Workflow пишется на чистом JavaScript. Основные конструкции — это функции, которые описывают шаги сценария.

Структура файла workflow

// .workflow.js — основной файл workflow export const meta = { name: 'code-review-workflow', description: 'Комплексное ревью кода по 5 измерениям', phases: ['review', 'synthesize'] }; export default async function(context) { // Описываем сценарий }

Основные функции

Функция Что делает
agent(name, prompt) Запускает конкретного агента с промптом. Возвращает ответ.
parallel([...tasks]) Запускает несколько задач параллельно. Ждёт все.
pipeline([...steps]) Конвейер: результат каждого шага передаётся следующему.
phase(name, fn) Объявляет фазу workflow. Для логирования и навигации.
log(message) Выводит сообщение в лог workflow. Для трассировки.

Пример: Fan-out workflow

export default async function(context) { await phase('review', async () => { const reviews = await parallel([ agent('innovator', 'Проверь архитектуру: ...'), agent('pragmatist', 'Проверь производительность: ...'), agent('conservative', 'Проверь безопасность: ...'), ]); log('Все ревью получены, синтезирую...'); return reviews; }); await phase('synthesize', async () => { return agent('writer-agent', 'Собери единый отчёт из ревью: ...'); }); }
⚠️ Важно: Workflow пишется на JavaScript, не TypeScript. Типы не нужны — это скрипт-оркестратор, а не библиотека.

💯 Когда это нужно

🔍 Код-ревью по измерениям

Код проверяют 5 агентов параллельно: безопасность, производительность, читаемость, тестопригодность, архитектура. Fan-out → синтез.

🏗️ Архитектурные дебаты

Инноватор, прагматик и консерватор спорят. Два раунда — сначала каждый со своей позицией, потом все видят чужие ответы. Adversarial.

🔄 Миграция кода

Один агент находит все файлы для миграции → Fan-out на N агентов (по файлу или группе) → каждый мигрирует → Loop проверяет что ничего не сломалось.

🛠️ Рефакторинг

Исследователь находит запахи кода → архитектор предлагает план → Fan-out: каждый агент рефакторит свой кусок → тестер проверяет → Loop пока тесты не зелёные.

Правило большого пальца

Если задача укладывается в один контекст одного агента — workflow не нужен. Если задача большая (много файлов, много измерений, много раундов обсуждения) или требует разных компетенций — workflow экономит время и делает результат качественнее.

🚀 Связь с Background Tasks

Workflow может использовать Background Tasks как часть сценария. Это даёт дополнительный параллелизм.

Workflow + Background Tasks

Пока основной workflow выполняет фазу ревью — в фоне может крутитьсявторостепенныйная задача: проверка CI, анализ зависимостей, генерация доки. Результаты фоновых задач подключаются к workflow на нужной фазе.

Схема взаимодействия

# Workflow выполняет основную задачу await phase('review', async () => { const reviews = await parallel([...]); return reviews; }); # Параллельно — фоновая задача (вне workflow) # claude --bg "проверь CI пока workflow крутится" # Workflow продолжает синтез await phase('synthesize', async () => { return agent('writer-agent', 'Собери отчёт...'); });
💡 Но: Workflow и Background Tasks — это разные механизмы. Workflow — это декларативный сценарий внутри сессии. Background Tasks — это фоновые задачи, которые можно запустить отдельно. Они хорошо дружат, но не обязательно использовать оба.

Итого: Workflow — это скрипт-оркестратор, который автоматизирует мульти-агентные сценарии. Вместо ручного запуска агентов и копирования контекста — один файл .workflow.js описывает весь конвейер. Паттерны: Pipeline, Fan-out, Adversarial, Loop-until-dry. Когда задача большая или требует разных компетенций — workflow экономит время и делает результат лучше. 🏭
