🔑 Что такое Permissions?

Claude Code — это агент, который выполняет реальные действия: читает файлы, пишет код, запускает команды, удаляет данные. Не каждое действие безопасно, поэтому нужна система, которая решает: можно это делать или нельзя.

Permissions — это встроенная система правил доступа Claude Code. Она определяет, какие инструменты (tools) агент может использовать без вопросов, какие — только с твоим подтверждением, а какие — вообще запрещены.

Проще простого: Permissions — это список «можно / нельзя / спросить» для каждого типа действия. Настройки живут в settings.json и работают на нескольких уровнях: глобальном, проектном и сессионном.

💡 Permissions и режимы работы (defaultMode) — это две разные системы, которые работают вместе. Режим задаёт общую политику, а правила permissions — конкретные исключения. Подробнее о режимах — в статье Режимы работы.

🚦 Три значения: allow, deny, ask

Каждое правило permissions имеет одно из трёх значений:

allow Разрешено

Инструмент выполняется молча, без подтверждения. Агент просто делает свою работу и идёт дальше.

Пример: чтение файлов, запуск тестов, поиск по коду — вещи, которые ты доверяешь агенту на 100%.

deny Запрещено

Инструмент заблокирован полностью. Агент не может его вызвать, даже если очень хочет. Не спрашивает — просто не работает.

Пример: вредоносные команды, нежелательные агенты, опасные операции, которые ты точно не хочешь видеть.

ask Спросить

Перед выполнением агент показывает тебе запрос на подтверждение. Ты видишь что именно будет сделано и решаешь: «ОК» или «Нет».

Пример: редактирование файлов, запуск bash-команд, использование MCP-инструментов — всё, что требует контроля.

📝 Синтаксис правил

Правила записываются в виде строк в массивах allow, deny или используются в настройках проекта. Синтаксис зависит от типа инструмента:

Точное совпадение

Bash(python3 script.py) Разрешает только эту конкретную команду
Read(/home/user/config.json) Разрешает чтение только этого файла
Skill(deep-research) Разрешает только этот скилл

Wildcard (шаблон с *)

Bash(git *) Любая команда, начинающаяся с git
Bash(python *) Любой вызов python
Bash(/home/andrei/mcp/*/server.py) Wildcard на уровне каталога
⚠️ Wildcard * означает всё что угодно после пробела или конца строки. Bash(git *) совпадёт с git status, git commit -m "msg", но не с github-cli status.

Только инструмент (без аргументов)

WebSearch Разрешает все вызовы веб-поиска
WebFetch Разрешает все вызовы получения веб-страниц

MCP-инструменты

mcp__poisk-mcp__search_web Инструмент search_web из MCP-сервера poisk-mcp
mcp__poisk-mcp__read_webpage Инструмент read_webpage из того же сервера

Формат MCP-инструментов: mcp__{server-name}__{tool-name}. Два подчёркивания между частями — это разделитель.

Встроенные инструменты Claude Code

Префикс Инструмент Пример правила
Bash(...) Запуск команд Bash(git *), Bash(python3 *)
Read(...) Чтение файлов Read(Grep *), Read(Glob *)
Edit(...) Редактирование файлов Edit(Write *), Edit(Edit *)
Write Создание файлов Write (все файлы)
WebSearch Веб-поиск WebSearch

⚖️ Приоритеты: deny > ask > allow

Правила проверяются в строгом порядке. Если одно и то же действие совпадает с правилами из разных массивов — побеждает правило с наивысшим приоритетом:

Порядок приоритетов

  1. deny — если действие запрещено, оно всегда будет заблокировано, даже если есть правило allow для того же инструмента
  2. ask — если не запрещено, проверяется: нужно ли спросить подтверждение
  3. allow — если не запрещено и не требует подтверждения — выполняется молча

Пример конфликта

{ "allow": ["Bash(rm *)"], "deny": ["Bash(rm -rf *)"] }

Результат:

  • rm file.txt — allow (совпадает с Bash(rm *), нет в deny)
  • rm -rf / — deny (совпадает с Bash(rm -rf *), deny побеждает)

Пошагово:

  1. Проверка режима. Если режим plan — пишущие инструменты заблокированы. Если bypassPermissions — всё разрешено без вопросов.
  2. Проверка deny. Если инструмент (с учётом аргументов) совпадает с любым правилом в deny — сразу блокируется.
  3. Проверка allow. Если есть совпадение в allow — выполняется молча.
  4. Ask по умолчанию. Если ничего не совпало — агент спрашивает у тебя.

⚙️ Конфигурация: settings.json

Правила permissions задаются в файле settings.json. Он существует на трёх уровнях:

Уровень Путь Когда применяется
Глобальный ~/.claude/settings.json Во всех проектах
Проектный .claude/settings.json (в корне проекта) Только в этом проекте
Локальный .claude/settings.local.json (в корне проекта) Личные настройки, не коммитятся в git

Пример полного конфига

// ~/.claude/settings.json — глобальные настройки { "permissions": { "allow": [ "Bash(git *)", "Bash(python *)", "Bash(python3 *)", "Bash(uv *)", "Bash(pytest *)", "Bash(npm *)", "Read(Grep *)", "Read(Glob *)", "Edit(Write *)", "Edit(Edit *)", "Skill(deep-research)", "WebSearch", "mcp__poisk-mcp__search_web" ], "deny": [ "Agent(explore)", "Agent(statusline-setup)", "WebFetch" ], "defaultMode": "auto" }, "model": "claude-go-deepseek-v4-flash", "language": "russian" }

Пример проектного конфига

// .claude/settings.json — настройки проекта { "permissions": { "allow": [ "Bash(npm run test *)", "Bash(npm run lint *)", "Bash(npx prettier *)" ], "deny": [ "Bash(rm -rf *)", "Bash(sudo *)" ] } }
💡 settings.local.json — личные настройки, которые не коммитятся в git. Используй для приватных правил: MCP-серверов, персональных путей, хуков. А settings.json в корне проекта — для командных правил.

🎛️ Режимы defaultMode

Поле defaultMode в permissions задаёт общую политику по умолчанию. Правила в allow/deny работают как исключения поверх этой политики.

Режим Чтение Запись Команды Для кого
default Спрашивает Спрашивает Спрашивает Новички, первый запуск
auto Молча Спрашивает Спрашивает Повседневная работа
acceptEdits Молча Молча Спрашивает Массовые правки
plan Молча Запрещено Запрещено Исследование, планирование
bypassPermissions Молча Молча Молча Опытные, максимальная скорость

Как правила и режимы работают вместе

Допустим, у тебя defaultMode: "auto" и правило:

"deny": ["Bash(rm *)"]

В режиме auto команды обычно спрашивают подтверждение. Но rm даже не дойдёт до вопроса — он будет заблокирован правилом deny.

А если добавить "Bash(rm *)" → allow, то в режиме auto команда rm будет выполняться молча, хотя обычно такие команды спрашивают. Правило allow перекрывает поведение режима.

✅ Best practices и типичные ошибки

Делай

  • Начинай с default или auto — расширяй по мере понимания
  • Используй deny для конкретных опасных команд: rm -rf, sudo
  • Прописывай wildcard широких категорий: Bash(git *), Bash(python *)
  • Для MCP-инструментов указывай конкретные имена: mcp__server__tool
  • Используй settings.local.json для приватных правил

Не делай

  • Не ставь bypassPermissions без понимания последствий
  • Не коммить settings.local.json — там приватные пути и ключи
  • Не путай allow с defaultMode — это разные механизмы
  • Не ставь слишком широкие allow без deny: Bash(*) опасно
  • Не забывай: deny побеждает allow, всегда

Типичные ошибки

Ошибка 1: Слишком широкий wildcard

"allow": ["Bash(*)"] // ОПАСНО! Разрешает ВСЁ

Проблема: Агент сможет запустить любую команду без вопросов — rm -rf /, curl ... | bash, что угодно.

Решение: Перечисли конкретные категории: Bash(git *), Bash(python *), Bash(npm *).

Ошибка 2: Путаница allow и defaultMode

{ "allow": ["Bash(rm *)"], "defaultMode": "auto" }

Ожидание: «auto спрашивает команды, значит rm будет спрашивать».

Реальность: Правило allow перекрывает поведение режима. rm выполнится молча.

Ошибка 3: Забыть про deny

{ "allow": ["Bash(rm *)"], // Нет deny для rm -rf! }

Проблема: rm file.txt и rm -rf / оба совпадают с Bash(rm *). Агент удалит всё без вопросов.

Решение: Всегда добавляй deny для опасных вариантов:

"deny": ["Bash(rm -rf *)"]

Ошибка 4: Неправильный синтаксис MCP

"allow": [ "mcp__poisk-mcp-search_web", // ОШИБКА: один _ вместо двух "mcp_poisk_mcp__search_web" // ОШИБКА: подчёркивания вместо дефисов ]

Правильно:

"allow": ["mcp__poisk-mcp__search_web"]

Дефис в имени сервера сохраняется. Разделитель — __ (двойное подчёркивание).

Итого: Permissions — это система «можно / нельзя / спросить» для каждого действия. Правила настраиваются через settings.json, работают с приоритетом deny > ask > allow, а defaultMode задаёт общую политику. Начни с auto, добавляй allow для рутины, deny для опасного, и не забывай про проектные настройки.
