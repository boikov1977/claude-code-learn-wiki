🏭 Что такое Workflow Tool

Если базовая статья Workflow объясняет зачем оркестрировать, эта — про то, как. Workflow-скрипт — это не просто «запусти агента и жди». Это детерминированная программа, которая:

  • Описывает точный порядок запуска десятков субагентов
  • Передаёт результаты между стадиями автоматически
  • Поддерживает параллелизм и барьеры синхронизации
  • Возобновляема после падения — если сессия упала, можно продолжить с того же места

Пишется на чистом JavaScript (не TypeScript!). Это не библиотека — это скрипт-оркестратор. Файл живёт с расширением .workflow.js.

Ключевое отличие от обычного агента: Обычный вызов агента — это запрос-ответ. Workflow — это программа с условиями, циклами, параллелизмом и управлением бюджетом. Агент не знает, что будет дальше. Workflow знает.

📐 Анатомия workflow-скрипта

Каждый workflow-скрипт состоит из двух частей: метаданные и тело.

Минимальный скрипт

// review.workflow.js export const meta = { name: 'code-review', description: 'Комплексное ревью кода с верификацией', phases: [ 'Review', 'Verify', 'Report' ] }; // Тело скрипта — здесь происходит магия await phase('Review'); const findings = await parallel( DIMENSIONS.map(dim => () => agent(`Проверь код по измерению «${dim}»`, { label: `Review: ${dim}`, schema: FINDING_SCHEMA }) ) ); await phase('Verify'); const verified = await pipeline( findings.filter(Boolean), finding => agent(`Опровергни: ${finding}`, { label: `Verify: ${finding.title}`, schema: VERDICT_SCHEMA }) ); await phase('Report'); return agent('Собери финальный отчёт', { schema: REPORT_SCHEMA });

meta — обязательные поля

Поле Тип Обязательно Описание
meta.name string Да Уникальное имя workflow. Используется в логах и для resume.
meta.description string Да Что делает workflow. Показывается в UI.
meta.phases string[] Да Список фаз. Каждый phase() в теле ссылается на имя из этого массива.

meta должен быть чистым литералом! Никаких переменных, функций, вычислений. Только строковые и числовые литералы, массивы литералов и объекты с литеральными значениями. Движок workflow десериализует meta отдельно от тела скрипта — если там будет process.env.X, он не сможет прочитать.

Функции, доступные в теле скрипта

Функция Сигнатура Описание
phase() phase(name) Объявляет текущую фазу. Для логов, навигации и resume.
agent() agent(prompt, opts?) Запускает субагента. Возвращает текст или schema-объект.
pipeline() pipeline(items, ...fns) Конвейер без барьера. Каждый item обрабатывается независимо.
parallel() parallel(fns) Параллельный запуск с барьером. Ждёт все результаты.
budget budget.total / .spent() / .remaining() Управление лимитом токенов.
return — Возвращает финальный результат workflow.

🤖 agent() — основной строительный блок

agent(prompt, opts?) — это функция, которая запускает одного субагента и возвращает результат. Это атом workflow: каждый вызов — отдельный LLM-запрос с собственным контекстом.

Параметры opts

Параметр Тип Описание
label string Отображаемое имя в прогресс-баре. Для читаемых логов: 'Review: безопасность'.
schema JSON Schema Структурированный вывод! LLM обязан вернуть объект, соответствующей схеме.
model string Переопределение модели для этого вызова. Например, 'claude-sonnet-4-20250514'.
effort string Уровень усилий: 'low', 'medium', 'high'.
isolation 'worktree' Git Worktree для изоляции мутаций файла. Каждый агент работает в своей копии.

Structured Output с schema

Когда передаёшь schema, агент обязан вернуть JSON-объект, соответствующий схеме. Это главный инструмент качества — вместо парсинга текста ты получаешь проверенную структуру данных.

// Определяем схему const FINDING_SCHEMA = { type: 'object', properties: { title: { type: 'string', description: 'Краткое описание' }, severity: { type: 'string', enum: ['low', 'medium', 'high', 'critical'] }, file: { type: 'string', description: 'Путь к файлу' }, line: { type: 'number', description: 'Номер строки' }, evidence: { type: 'string', description: 'Цитата из кода' } }, required: ['title', 'severity', 'file'] }; // Используем в agent() const finding = await agent( 'Найди баги в файле auth.js', { schema: FINDING_SCHEMA, label: 'Security review' } ); // finding — это объект: { title: '...', severity: 'high', file: '...', ... }

Обработка ошибок

Если агент упал (таймаут, ошибка API, невалидный JSON) или был пропущен — agent() возвращает null, а не бросает ошибку. Это сделано специально: workflow должен быть устойчив к сбоям отдельных агентов.

Правило: всегда фильтруй результаты.

const results = await parallel(tasks); const valid = results.filter(Boolean); // убираем null if (valid.length === 0) { log('Все агенты упали, workflow не может продолжить'); return { error: 'No valid results' }; }

⚔️ pipeline() против parallel()

Это ключевое архитектурное решение в каждом workflow. Неправильный выбор = лишние минуты ожидания или сломанная логика.

pipeline() — конвейер без барьера

По умолчанию. Каждый item проходит все стадии независимо.

Item A может быть в Stage 2, пока Item B ещё в Stage 1. Никто никого не ждёт.

Wall-clock время = время самого медленного item * количество стадий.

// Каждый файл ревьюится, потом сразу фиксится — без ожидания других const results = await pipeline( files, file => agent(`Review ${file}`, { label: `Review: ${file}`, schema: REVIEW_SCHEMA }), review => agent(`Fix: ${review.evidence}`, { label: `Fix: ${review.file}`, isolation: 'worktree' }) ); // Item A уже в Fix, пока B ещё в Review — это нормально!

parallel() — барьер синхронизации

Ждёт ВСЕ результаты перед переходом к следующей стадии.

Используй только когда стадия N требует все результаты стадии N-1.

// Все ревью должны быть готовы, чтобы выбрать лучшее const reviews = await parallel( approaches.map(a => () => agent(a.prompt, { label: `Approach: ${a.name}` }) ) ); // Барьер: все reviews получены const best = await agent( `Выбери лучшее из ${reviews.length} подходов`, { schema: BEST_APPROACH_SCHEMA } );

Когда нужен барьер

Сценарий Барьер нужен? Почему
Дедупликация: собрали все findings, убираем дубли Да Нужно видеть все находки, чтобы сравнить
Early exit: «0 bugs found — нечего верифицировать» Да Нужно знать общее количество до принятия решения
Сравнение: «найди лучшее из всех решений» Да Агент-судья должен видеть все варианты
Просто собрать результаты Нет pipeline(items, ..., r => r.flat()) внутри стадии
Стадии концептуально разные Нет Для этого pipeline() и существует

❌ Ошибка новичков: барьер без необходимости

«Надо просто собрать результаты в массив» — это не причина для parallel().

// ЛИШНИЙ БАРЬЕР — все файлы ждут самый медленный const results = await parallel( files.map(f => () => agent(`Process ${f}`)) ); // Файл A обработался за 5с, файл B за 5 минут. // Все ждут 5 минут. // ПРАВИЛЬНО — pipeline, каждый работает в своём темпе const results = await pipeline( files, f => agent(`Process ${f}`) );

Барьерное ожидание = лишние минуты. Каждый барьер замедляет workflow на время самого медленного агента.

🔍 Паттерны верификации

LLM могут галлюцинировать, выдумывать баги и ошибочно определять проблемы. Workflow Tool позволяет строить системы верификации, которые ловят эти ошибки. Вот три проверенных паттерна.

1. Adversarial Verification — аверсариал

Запускаешь N скептиков, каждый пытается опровергнуть finding. По умолчанию «refuted=true»: скептик должен найти конкретную причину, почему находка невалидна.

flowchart TD FINDING["Находка:\n\"SQL injection в auth.js:42\""] FINDING --> S1["Скептик 1"] FINDING --> S2["Скептик 2"] FINDING --> S3["Скептик 3"] S1 --> V1["refuted: false\n(подтверждает)"] S2 --> V2["refuted: true\n(параметризованный запрос)"] S3 --> V3["refuted: false\n(уязвимость реальна)"] V1 --> VOTE["Голосование: 2/3\nне опровергли"] V2 --> VOTE V3 --> VOTE VOTE -->|"2+ не опровергли"| PASS["Находка подтверждена"] VOTE -->|"2+ опровергли"| KILL["Находка отброшена"] style FINDING fill:#fff8f0,stroke:#e36209,stroke-width:2px style PASS fill:#f6fff8,stroke:#28a745,stroke-width:2px style KILL fill:#fff5f5,stroke:#d73a49,stroke-width:2px style VOTE fill:#f1f8ff,stroke:#0366d6,stroke-width:2px
const VERDICT_SCHEMA = { type: 'object', properties: { refuted: { type: 'boolean' }, reason: { type: 'string' } }, required: ['refuted', 'reason'] }; async function adversarialVerify(claim, n = 3) { const votes = await parallel( Array.from({ length: n }, () => () => agent( `Опровергни: ${claim}. По умолчанию refuted=true, если не уверен.`, { schema: VERDICT_SCHEMA, label: 'Skeptic' } ) ) ); const valid = votes.filter(Boolean); const survives = valid.filter(v => !v.refuted).length >= Math.ceil(n / 2); return { survives, votes: valid }; }

Почему «по умолчанию refuted=true»? LLM по умолчанию склонен соглашаться. Если просто спросить «это баг?» — большинство скажет «да». Фраза «по умолчанию refuted=true» переворачивает модель: теперь скептик должен доказать валидность, а не просто подтвердить. Это снижает false positive на 40-60%.

2. Judge Panel — панель судей

N независимых подходов к одной задаче, потом один агент синтезирует лучшее. Используй когда нужно выбрать из нескольких вариантов, а не просто отфильтровать.

const APPROACHES = [ { name: 'aggressive', prompt: 'Предложи максимально агрессивное решение' }, { name: 'conservative', prompt: 'Предложи максимально безопасное решение' }, { name: 'balanced', prompt: 'Предложи сбалансированное решение' }, ]; const approaches = await parallel( APPROACHES.map(a => () => agent(a.prompt, { label: `Judge: ${a.name}` })) ); const synthesis = await agent( `Синтезируй из ${approaches.length} подходов лучшее решение. ` + `Возьми сильные стороны каждого.`, { schema: SYNTHESIS_SCHEMA, label: 'Synthesis' } );

3. Loop-until-dry — цикл до иссякания

Повторяешь проверку, пока новые находки не иссякнут. Паттерн «dry count»: считаешь количество пустых итераций подряд.

let dry = 0; const allFindings = []; while (dry < 2) { const found = await agent( 'Найди баги в коде. Если ничего не найдено — верни пустой массив.', { schema: BUGS_SCHEMA, label: `Scan #${allFindings.length}` } ); if (!found.bugs || found.bugs.length === 0) { dry++; log(`Пустой проход ${dry}/2`); continue; } dry = 0; // сброс счётчика — нашлись новые баги allFindings.push(...found.bugs); log(`Найдено ${found.bugs.length} багов (всего: ${allFindings.length})`); } // Два пустых прохода подряд —останавливаемся
💡 Зачем 2, а не 1? Один пустой проход может быть случайностью (LLM не заметил что-то). Два пустых прохода подряд — статистически надёжный сигнал, что находки иссякли.

💰 Budget management

Workflow Tool имеет встроенный механизм управления лимитом токенов. Это критично для длительных workflow с десятками агентов.

API

Свойство Возвращает Описание
budget.total number | null Лимит на весь workflow. null = без лимита.
budget.spent() number Сколько токенов уже израсходовано.
budget.remaining() number Сколько осталось. Infinity если нет лимита.

Пример: цикл с бюджетом

// Ограничиваем верификацию 50K токенов const findings = [/* ... */]; const verified = []; for (const f of findings) { if (budget.total && budget.remaining() < 50_000) { log(`Бюджет на исходе: ${budget.remaining()} токенов. Стоп.`); break; } const verdict = await adversarialVerify(f); if (verdict.survives) verified.push(f); } log(`Потрачено: ${budget.spent()}, осталось: ${budget.remaining()}`);

Лимит жёсткий. При превышении budget.total вызов agent() выбрасывает ошибку. Workflow может обработать это через try/catch, но остановка неизбежна. Планируй бюджет заранее!

💡 Эвристика: Один вызов agent() средней сложности сжигает ~5-15K токенов. Workflow с 3 фазами по 5 агентов = ~150K токенов. Ставь лимит с запасом 30%.

⚠️ Когда Workflow — overkill

Workflow Tool мощный, но не всегда нужен. Вот когда не стоит его использовать.

✅ Достаточно простого агента

  • Простой поиск/ответ — один agent()
  • Разовый вопрос — обычный разговор с Claude
  • «Проверь CI» — /loop
  • Ревью одного файла — Agent tool

🚀 Workflow нужен

  • 5+ агентов с разными ролями
  • Сложный контроль потока (условия, циклы)
  • Детерминированность и воспроизводимость
  • Возобновляемость после падения сессии
  • Управление бюджетом токенов

Правило большого пальца

Если задача укладывается в один контекст одного агента — workflow не нужен. Если задача требует 3+ агентов, разных компетенций или итеративной верификации — workflow экономит время и делает результат качественнее.

🚀 Практический пример: полный скрипт

Рабочий workflow для поиска багов с верификацией. 3 фазы, schema-валидация, adversarial verify.

// bug-hunt.workflow.js // Полный workflow: поиск багов → верификация → отчёт export const meta = { name: 'bug-hunt', description: 'Мульти-измерительный поиск багов с adversarial верификацией', phases: [ 'Scan', 'Verify', 'Report' ] }; // ── Схемы ── const BUG_SCHEMA = { type: 'object', properties: { title: { type: 'string' }, severity: { type: 'string', enum: ['low', 'medium', 'high', 'critical'] }, file: { type: 'string' }, line: { type: 'number' }, evidence: { type: 'string' }, suggestion: { type: 'string' } }, required: ['title', 'severity', 'file'] }; const BUGS_SCHEMA = { type: 'object', properties: { bugs: { type: 'array', items: BUG_SCHEMA } }, required: ['bugs'] }; const VERDICT_SCHEMA = { type: 'object', properties: { refuted: { type: 'boolean' }, reason: { type: 'string' } }, required: ['refuted', 'reason'] }; const REPORT_SCHEMA = { type: 'object', properties: { summary: { type: 'string' }, bugs: { type: 'array', items: BUG_SCHEMA }, total_found: { type: 'number' }, total_survived: { type: 'number' } }, required: ['summary', 'bugs', 'total_found', 'total_survived'] }; // ── Измерения сканирования ── const DIMENSIONS = [ { name: 'security', prompt: 'Ищи SQL injection, XSS, path traversal, hardcoded secrets' }, { name: 'logic', prompt: 'Ищи off-by-one, race conditions, unhandled edge cases' }, { name: 'reliability', prompt: 'Ищи uncaught exceptions, missing null checks, resource leaks' } ]; // ── Тело workflow ── // ФАЗА 1: Параллельное сканирование по 3 измерениям await phase('Scan'); const scanResults = await parallel( DIMENSIONS.map(dim => () => agent( `Ты — ${dim.name} эксперт. ${dim.prompt}\n\n` + 'Проанализируй весь код в проекте и найди проблемы.', { label: `Scan: ${dim.name}`, schema: BUGS_SCHEMA, effort: 'high' } ) ) ); // Собираем все баги, дедуплицируем по file+line const allBugs = scanResults .filter(Boolean) .flatMap(r => r.bugs) .filter((bug, i, arr) => arr.findIndex(b => b.file === bug.file && b.line === bug.line) === i ); log(`Найдено ${allBugs.length} уникальных багов после дедупликации`); if (allBugs.length === 0) { return { summary: 'Багов не найдено!', bugs: [], total_found: 0, total_survived: 0 }; } // ФАЗА 2: Adversarial верификация каждого бага await phase('Verify'); const verified = await pipeline( allBugs, bug => parallel( Array.from({ length: 3 }, () => () => agent( `Опровергни этот баг. По умолчанию refuted=true.\n` + `Баг: ${bug.title}\nФайл: ${bug.file}:${bug.line}\n` + `Доказательство: ${bug.evidence}\n` + `Если НЕ можешь найти конкретную причину опровержения — ` + `ставь refuted=false.`, { schema: VERDICT_SCHEMA, label: `Verify: ${bug.title}` } ) ), votes => { const valid = votes.filter(Boolean); const survives = valid.filter(v => !v.refuted).length >= 2; return survives ? bug : null; } ) ); const survivors = verified.filter(Boolean); log(`Верификация: ${survivors.length}/${allBugs.length} багов выжили`); // ФАЗА 3: Синтез отчёта await phase('Report'); return agent( `Составь отчёт о найденных багах.\n` + `Всего найдено: ${allBugs.length}\n` + `Подтверждено: ${survivors.length}\n\n` + `Данные:\n${JSON.stringify(survivors, null, 2)}`, { schema: REPORT_SCHEMA, label: 'Final Report' } );

💡 Заметь: Верификация использует pipeline(), а не parallel(). Каждый баг верифицируется независимо — зачем ждать, пока скептики проверят Bug #1, чтобы начать проверку Bug #5? Барьер нужен только перед фазой Report.

Итого: Workflow Tool — это детерминированная программа для оркестрации десятков субагентов. Ключевые инструменты: agent() со schema, pipeline() vs parallel() (выбирай осознанно!), adversarial verification для качества, budget management для контроля затрат.
