Skill vs Slash Command в Claude Code

? Зачем это знать?

Claude Code имеет два встроенных механизма для автоматизации — Slash Commands и Skills. Оба позволяют создавать повторно используемые инструкции, но работают принципиально по-разному. Понимание разницы помогает выбрать правильный инструмент для задачи и избежать situations, когда простая команда перегружена логикой, или сложный workflow пытается втиснуться в однострочный промпт.

Ключевая мысль: Slash Command — это одноразовый промпт, который Claude выполняет целиком. Skill — это многошаговая инструкция, которая может планировать, задавать вопросы, ветвиться и автоматически запускаться при определённых условиях.

/ Slash Command — короткий промпт

Slash Command — это markdown-файл, который превращается в команду, вызываемую через /имя в чате Claude Code. По сути это готовый промпт: Claude читает файл и выполняет инструкцию.

Где лежат

  • .claude/commands/ — кастомные команды проекта
  • ~/.claude/commands/ — глобальные команды пользователя
  • В плагинах: plugins/<name>/commands/

Формат файла

--- allowed-tools: Bash(git add:*), Bash(git commit:*) description: Создать git-коммит --- ## Context - Current git status: !`git status` - Current git diff: !`git diff HEAD` ## Your task Based on the above changes, create a single git commit.

Ключевые особенности

  • Single-turn: одна команда = один ответ Claude. Без ветвления, без циклов.
  • Явный вызов: пользователь должен набрать /commit в чате.
  • Frontmatter: allowed-tools, description, argument-hint, model.
  • $ARGUMENTS: подстановка аргументов из командной строки.
  • !`...` (backtick): динамическое подключение вывода shell-команд прямо в промпт.
  • Размер: обычно 5-30 строк markdown.
💡 Slash Command — это не shell-команда терминала. Это команды, которые пишутся прямо в чате Claude Code. Набери / и увидишь автодополнение.

Примеры

/commit — быстрый коммит

--- allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*) description: Create a git commit --- ## Context - Current git status: !`git status` - Current git diff (staged and unstaged changes): !`git diff HEAD` ## Your task Based on the above changes, create a single git commit.

Простой, понятный, работает в одно касание.

/commit-push-pr — цепочка git-операций

--- allowed-tools: Bash(git checkout:*), Bash(git add:*), Bash(git commit:*), Bash(git push:*), Bash(gh pr create:*) description: Commit, push, and open a PR --- ## Your task 1. Create a new branch if on main 2. Create a single commit 3. Push the branch to origin 4. Create a pull request using `gh pr create`

Всё ещё single-turn, но с цепочкой шагов.

⚡ Skill — многошаговая инструкция

Skill — это структурированная инструкция, которая living в отдельной директории (skills/<name>/SKILL.md). Она может быть многошаговой, ветвящейся и автоматически запускаемой.

Где лежат

  • ~/.claude/skills/<name>/SKILL.md — кастомные скиллы пользователя
  • .claude/skills/<name>/SKILL.md — скиллы проекта
  • В плагинах: plugins/<name>/skills/<name>/SKILL.md

Структура директории

openspec-apply-change/ ├── SKILL.md # Обязательный файл ├── scripts/ # Исполняемые скрипты (опционально) ├── references/ # Документация для загрузки по необходимости └── assets/ # Шаблоны, иконки, ресурсы

Формат SKILL.md

--- name: openspec-new-change description: Start a new OpenSpec change. Use when the user wants to create a new feature, fix, or modification with structured workflow. license: MIT compatibility: Requires openspec CLI. metadata: author: openspec version: "1.0" --- Start a new change using the experimental artifact-driven approach. **Input**: The user's request should include a change name or description. **Steps** 1. **If no clear input provided, ask what they want to build** Use the **AskUserQuestion tool** to ask the user. 2. **Determine the workflow schema** Use the default schema unless user explicitly requests a different one. 3. **Create the change directory** ```bash openspec new change "<name>" ``` 4. **Show the artifact status** ```bash openspec status --change "<name>" --json ``` 5. **STOP and wait for user direction** **Guardrails** - Do NOT create any artifacts yet - Do NOT advance beyond showing the first artifact template

Ключевые особенности

  • Multi-step: может содержать десятки шагов с ветвлениями.
  • Автотриггер: Claude автоматически выбирает скилл по description, если пользователь не вызвал явно.
  • Progressive Disclosure: метаданные (~100 слов) загружаются всегда, тело — при триггере, ресурсы — по необходимости.
  • Guardrails: может запрещать определённые действия (например, "никогда не пиши код").
  • Бандлы ресурсов: скрипты, документация, шаблоны — всё в одной папке.
  • Размер: от 20 до 500+ строк markdown.

Примеры скиллов

openspec-explore — исследование кодовой базы

Весьма нетривиальный скилл: проверяет контекст, определяет активные изменения, предлагает задокументировать инсайты. Содержит ASCII-диаграммы, таблицы, guardrails.

--- name: openspec-explore description: Enter explore mode - a thinking partner for exploring ideas, investigating problems, and clarifying requirements. --- Enter explore mode. Think deeply. Visualize freely. **IMPORTANT: Explore mode is for thinking, not implementing.** You may read files and investigate the codebase, but NEVER write code. **This is a stance, not a workflow.**

playground — создание интерактивных HTML-площадок

Определяет тип playground, загружает нужный шаблон из templates/, строит интерактивный HTML. Автотриггер: "сделай playground для X".

📐 Как это работает: блок-схема

Ключевое различие — в жизненном цикле. Slash Command выполняется один раз. Skill может циклить, ветвиться и автоматически активироваться.

Главная разница: Slash Command — это функция (вход → выход). Skill — это программа (условия, циклы, вопросы, guardrails).

Примечание о наследии (legacy)

Раньше скиллы тоже клались в commands/. Сейчас предпочтительный формат — skills/<name>/SKILL.md. Оба формата загружаются одинаково, но directory-based удобнее: он позволяет хранить скрипты, шаблоны и справочники рядом с инструкцией.

🔬 Практические примеры

Когда выбрать Slash Command

/commit

Простая операция: посмотреть git status, сделать add + commit. Один turn, без вопросов.

# .claude/commands/commit.md --- allowed-tools: Bash(git add:*), Bash(git commit:*) description: Создать git-коммит из текущих изменений --- ## Context - Current git status: !`git status` - Current git diff: !`git diff HEAD` ## Your task Stage all changes and create a single commit with a descriptive message.

/format

Запустить форматирование кода. Одна команда, один результат.

# .claude/commands/format.md --- allowed-tools: Bash(npx prettier:*), Bash(npx eslint:*) description: Отформатировать код проекта --- Отформатируй весь код в проекте с помощью prettier, затем исправь ошибки eslint. Покажи что изменилось.

Когда выбрать Skill

openspec-apply-change

Многошаговый workflow: выбор change, проверка статуса, чтение контекста, реализация задач по одной, обработка ошибок, паузы для уточнений. Невозможно описать в single-turn.

playground

Определяет тип playground из запроса, выбирает шаблон, строит HTML, открывает в браузере. Автотриггер: "сделай playground для стилей CSS" — Claude сам подхватит без /-вызова.

✅ Best Practices

Для Slash Commands

  • Держи короткими: 5-30 строк. Если больше — возможно, нужен Skill.
  • Используй !`...` для динамического контекста (git status, diff и т.д.).
  • Указывай allowed-tools — это убирает лишние запросы подтверждения.
  • Описывай description чётко — это то, что видит пользователь в подсказках.
  • Используй $ARGUMENTS когда нужен ввод от пользователя.

Для Skills

  • Пиши description "агрессивно": Claude склонен "недотриггерить" скиллы. Указывай конкретные контексты и фразы, при которых скилл должен сработать.
  • Держи SKILL.md под 500 строк: если больше — выноси справочники в references/.
  • Используй Progressive Disclosure: метаданные (~100 слов) всегда в контексте, тело загружается при триггере, ресурсы — по необходимости.
  • Добавляй Guardrails: явно запрещай то, что скилл не должен делать.
  • Объясняй "почему": LLM лучше понимает инструкции с обоснованием, чем жёсткие MUST/NEVER.
  • Тестируй: создавай 2-3 тестовых промпта и проверяй, что скилл срабатывает корректно.

⚠️ Типичные ошибки

1. Slash Command на 200 строк

Команда /deploy превратилась в монстра с 15 шагами, ветвлениями и обработкой ошибок. Claude пытается выполнить всё за один turn и теряет контекст.

Решение: Разбить на Skill с шагами и паузами.

2. Skill без guardrails

Скилл my-deployer не запрещает определённые действия. Claude при удалении деплоя случайно удаляет production-данные.

Решение: Всегда добавляй guardrails: "Do NOT touch production data", "Do NOT skip confirmation".

3. Пытаться сделать автотриггер для Slash Command

Slash Command не может срабатывать автоматически — только по /-вызову. Если нужна автоактивация — это Skill.

Решение: Используй Skill с описательным description.

4. Skill без структуры

SKILL.md содержит 500 строк сплошного текста без заголовков, шагов и разделителей. Claude не может эффективно "читать" такую инструкцию.

Решение: Используй заголовки, нумерованные шаги, маркированные списки, примеры кода.

5. Slash Command для задачи, требующей вопросов

/create-component не может спросить "как назвать компонент?" — это single-turn. Пользователь вынужден каждый раз указывать имя в $ARGUMENTS.

Решение: Skill с AskUserQuestion на шаге 1.

Сценарий Рекомендация Почему
Создать коммит из текущих изменений Slash Command Один turn, без вопросов, простой контекст
Запустить тесты и показать отчёт Slash Command Одна команда, один результат
Реализовать задачи из change-листа Skill Много шагов, нужно читать контекст, обрабатывать ошибки
Исследовать кодовую базу перед архитектурным решением Skill Ветвление, вопросы, визуализация, guardrails
Отформатировать код Slash Command Предсказуемая операция, не требует ветвления
Создать новый компонент с выбором типа Skill Нужно спросить тип, имя, позицию в архитектуре
Деплой с проверками Зависит от сложности Простой деплой — Slash, с проверками и откатом — Skill
⚡ Золотое правило: Если инструкция помещается в 30 строк без потери смысла — это Slash Command. Если нужна логика, вопросы, циклы или guardrails — это Skill.

Итого: Slash Command — для простых, предсказуемых операций в одно касание. Skill — для сложных workflow с логикой, вопросами и автоматическим триггером. Оба — встроенные механизмы Claude Code, ничего устанавливать не нужно.
