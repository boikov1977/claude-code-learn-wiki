📊 Три уровня сложности: от промпта до роя агентов

Microsoft Azure Architecture Center делит все AI-системы на три уровня сложности. Это полезная шкала — она помогает не прыгнуть сразу в multi-agent там, где хватит одного промпта.

Уровень Описание Когда использовать Пример
L1: Direct Model Call Одиночный промпт. Никакой логики агента, никаких инструментов Простые вопросы-ответы, суммаризация «Перепиши этот абзац»

💡 Ключевое правило: Начинай с L2. Переходи на L3 только когда single-agent не справляется. Multi-agent потребляет ~15x больше токенов — это не «бесплатное улучшение», а осознанный трейд-офф.

Источник: Microsoft Azure Architecture Center, «AI Agent Orchestration Patterns», May 2026. Подтверждено Princeton NLP: single-agent сравнялся или превзошёл multi-agent в 64% бенчмарков.

🏗️ Пять фундаментальных паттернов (Anthropic)

Anthropic в декабре 2024 опубликовали статью «Building Effective Agents», где выделили пять базовых workflow-паттернов. Это фундамент — все multi-agent системы собираются из этих блоков как из кирпичиков.

1️⃣ Prompt Chaining — линейная цепочка

Суть: Каждый LLM-вызов обрабатывает результат предыдущего. Как конвейер: шаг A → шаг B → шаг C.

Когда использоватьПримеры
Задачи с чёткими фиксированными шагами, где каждый шаг — ворота для следующего • Генерация тестов → прогон → отчёт
• Сборка → деплой на staging → smoke test → деплой в прод
• Checkstyle → юнит-тесты → интеграционные тесты

2️⃣ Routing — маршрутизация

Суть: Классификатор определяет тип задачи и направляет к специализированному обработчику.

Когда использоватьПримеры
Разные типы запросов требуют разных подходов • Тикеты: баг / фича / улучшение → разные агенты
• Вопросы:代码 / архитектура / процессы → разные эксперты
• Pull Request: bugfix / feature / refactoring → разные ревьюверы

Когда использоватьПримеры
Сложные задачи, где заранее неизвестно, сколько воркеров понадобится и какие именно • Реализация фичи: анализ → кодинг → тесты → документация
• Исследование: веерный поиск → чтение → верификация → отчёт
• Рефакторинг: поиск мест → правка → проверка совместимости
🔥 Это наш основной паттерн. Deep Research workflow (scout → read → verify → synthesize) и openspec-manager (propose → apply → sync → archive) — оба реализуют Orchestrator-Workers. Именно эту архитектуру использует Anthropic в своей production multi-agent research системе (Anthropic Engineering Blog).

5️⃣ Evaluator-Optimizer — генератор + оценщик в цикле

Суть: Один агент генерирует, другой оценивает. Цикл до достижения качества.

Когда использоватьПримеры
Задачи с множественными критериями качества, где нужна итерация • Архитектурные дебаты (у нас — innovator vs pragmatist vs conservative)
• Юридические документы — генерация → проверка → доработка
• Переводы — перевод → верификация носителем → корректировка
• Code review — код → ревью → исправление

☁️ Пять паттернов multi-agent оркестрации (Microsoft Azure)

Если Anthropic описывает фундаментальные строительные блоки, то Microsoft Azure — готовые архитектурные решения для multi-agent. Они строятся поверх тех же пяти паттернов.

Паттерн Azure Аналог у Anthropic Суть
Sequential Prompt Chaining Агенты в предопределённом линейном порядке. Пример: генерация контракта (шаблон → пункты → compliance → риски)
Concurrent Parallelization Параллельные агенты на одной задаче с агрегацией результатов
Group Chat — Общий чат с координатором. Для брейншторминга и human-in-the-loop
Handoff Routing Агенты динамически передают управление по контексту. Для customer support
Magentic Orchestrator-Workers Оркестратор с планированием (на базе AutoGen Magentic-One). Для сложных открытых задач
💡 Практический совет: Групповой чат (Group Chat) — единственный паттерн, который действительно сложен в реализации. Sequential и Concurrent — самые простые и надёжные. Начинай с них.

🎭 Парадокс коллаборации — AI в 60% работы, делегирование 0-20%

Anthropic Trends Report выявил интересную закономерность: разработчики используют AI в ~60% рабочего процесса, но полностью делегируют (без проверки каждой строки) лишь 0-20% задач.

Почему так? Это не недоверие к AI. Это осознанный выбор:

  • Код, написанный AI, требует верификации — как код джуниора
  • Встраивание проверок на каждом этапе даёт лучшие результаты, чем тотальное доверие
  • Самые успешные кейсы (Rakuten — 7 часов автономной работы с верификацией результата) используют активный контроль

Что это значит на практике

Проектируй человеческий контроль, а не полную автономию:

  • Встраивай точки проверки (review gates) на каждом этапе
  • Используй парадигму: Architect → Orchestrate → Test → Review (TeamDay.ai)
  • Агент делает черновик — человек утверждает направление — агент доделывает
  • Не пытайся автоматизировать всё — это путь к техдолгу

💰 Экономика и токены

Multi-agent системы потребляют примерно в 15 раз больше токенов, чем обычный чат (Anthropic Engineering Blog). Использование токенов объясняет 80% вариативности производительности в бенчмарках (BrowseComp).

Когда multi-agent оправдан экономически

  • Высокоценные задачи, где ошибка дороже токенов (архитектура, безопасность)
  • Задачи с активной параллелизацией (10 агентов за 5 минут дешевле, чем 1 агент за час)
  • Исследовательские задачи, где полнота важнее скорости (deep research)

Когда multi-agent НЕ оправдан

  • Простые задачи, которые решает один промпт
  • Задачи, где время ожидания важнее качества (перепиши этот текст)
  • Эксперименты «на всякий случай запущу 5 агентов»

Наша ситуация

Мы используем Deep Research (multi-agent workflow) для сложных исследований — и один сеанс отъедает ~13k токенов на субагентов (как показала практика). Для сравнения: обычная сессия — ~40k токенов на основной контекст. Multi-agent реально в 2-3 раза дороже, не в 15, потому что мы не гоним все 20 агентов последовательно — они работают параллельно в рамках одного workflow.

⏳ Долгоживущие агенты (Long-Lived Agents)

Anthropic Trends Report (тренд №5): горизонт задач агентов расширяется от минут до дней и недель. Агенты больше не делают одноразовые запросы — они живут в проекте.

🧩 Наша экосистема с точки зрения best practices

Давай честно сравним то, что у нас есть, с industry best practices.

Что мы делаем правильно ✅

  • Orchestrator-Workers — openspec-manager и deep-research workflow
  • Evaluator-Optimizer — архитектурные дебаты (innovator / pragmatist / conservative)
  • MCP isolation — серверы живут во frontmatter агентов, не засоряют контекст
  • Context через CLAUDE.md — качество контекста важнее промптов
  • Human-in-the-loop — permissionMode: auto с возможностью подтвердить или отклонить

Что можно улучшить 🔧

  • Нет тестирующего агента — генерация и прогон тестов вручную через Bash
  • Code Review — скилл, не агент — работает, но не имеет собственного контекста
  • Нет deploy-агента — деплой wiki делается через Bash с инструкциями
  • Нет commit-агента — коммиты пишутся вручную (или через меня)
  • Prompt Chaining не выделен как паттерн — потенциально: tester-agent

✅ Best Practices — сводка

✅ Делай

  • Начинай с Single Agent with Tools (L2)
  • Переходи на multi-agent только когда L2 не справляется
  • Используй Orchestrator-Workers для coding-задач
  • Инвестируй в качество CLAUDE.md и контекста
  • Проектируй review gates на каждом этапе
  • Изолируй MCP-серверы во frontmatter агентов
  • Измеряй cost-per-task, не только скорость

❌ Не делай

  • Не строй multi-agent «на всякий случай»
  • Не верь маркетингу «multi-agent — единственное будущее»
  • Не давай агентам полную автономию без guardrails
  • Не игнорируй стоимость токенов
  • Не микроменеджь агента — дай ему autonomy
  • Не пиши длинные промпты — лучше улучши CLAUDE.md
  • Не пытайся автоматизировать всё — 0-20% делегирования норма
