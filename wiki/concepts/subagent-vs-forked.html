🤖 Что такое субагенты

Оба типа субагентов — это встроенные механизмы Claude Code, которые позволяют вынести часть работы в отдельный контекст. Основной Claude вызывает субагента, тот работает, и возвращает результат. Главное преимущество — грязь остаётся в субагенте: прочитанные файлы, логи, промежуточные команды не засоряют основной контекст.

Ключевая идея: Субагент — это Claude в отдельном контекстном окне. Он имеет свой системный промпт, свою историю диалога и свои инструменты. Основной Claude получает только финальный результат.

Существует два варианта: обычный Subagent и Forked Subagent. Разница — в том, что попадает в контекст субагента при запуске.

📦 Subagent (Agent tool)

Обычный субагент запускается через инструмент Agent. Он получает чистый контекст — только системный промпт, описанный в frontmatter агента, и задачу, которую передал основной Claude.

Как работает

  • Основной Claude вызывает Agent с именем агента и задачей
  • Субагент стартует с чистого листа — без истории предыдущего разговора
  • Получает свой системный промпт из файла агента (~/.claude/agents/agent-name.md)
  • Имеет доступ к инструментам, описанным в tools секции frontmatter
  • Может использовать MCP-серверы, указанные в mcpServers
  • Возвращает только финальный ответ — основной Claude не видит промежуточные шаги
# Пример вызова обычного субагента Agent({ "agent": "explorer", "prompt": "Найди все функции, которые используют декоратор @cache в src/" }) # Результат: список функций (чистый, без мусора)

Плюсы

  • Минимальный расход токенов — в контекст субагента не копируется история
  • Чистый контекст — субагент видит только свою задачу, без отвлекающих факторов
  • Предсказуемость — субагент не зависит от того, что обсуждалось ранее
  • Изоляция — субагент не может «случайно» использовать информацию, которую не должен видеть

Минусы

  • Не знает контекст — если основной Claude уже принял решение, субагент о нём не узнает
  • Нужно формулировать задачу подробно — всю необходимую информацию нужно передать в prompt

🔀 Forked Subagent

Forked Subagent (форкнутый субагент) — это тот же механизм Agent, но с копией текущей истории диалога. Субагент стартует не с чистого листа, а с полной копией того, что было в основном контексте на момент вызова.

Как работает

  • Основной Claude вызывает Agent с параметром fork: true
  • Субагент получает копию всей истории диалога на момент вызова
  • Также получает свой системный промпт из файла агента
  • Все инструменты и MCP доступны так же, как и в обычном субагенте
  • Возвращает финальный ответ
# Пример вызова форкнутого субагента Agent({ "agent": "pragmatist", "prompt": "Пересмотри наш выбор архитектуры с точки зрения производительности", "fork": true }) # Субагент видит: все предыдущие решения, обсуждения, прочитанные файлы

Плюсы

  • Полный контекст — субагент знает всё, что обсуждалось ранее
  • Не нужно пересказывать — не нужно формулировать задачу с нуля
  • Идеально для ревью — агент-критик может оценить решение в полном контексте

Минусы

  • Высокий расход токенов — копируется вся история, включая прочитанные файлы и выводы команд
  • Контекст может быть загрязнён — субагент получает весь мусор предыдущей работы
  • Сложнее контролировать — субагент может «отвлечься» на детали, не относящиеся к задаче

💡 Когда какой использовать

Используй обычный Subagent когда:

  • Исследование кода — «найди все места, где используется X» (explorer)
  • Документирование — «напиши README для этого модуля» (writer-agent)
  • Узкая задача — «запусти тесты и верни список упавших»
  • Не нужен контекст — задача понятна без предыстории
  • Экономия токенов — история большая, а задача простая
# Типичные вызовы # Быстрый поиск Agent({ agent: "explorer", prompt: "Где определена функция parseConfig?" }) # Документирование Agent({ agent: "writer-agent", prompt: "Обнови CHANGELOG под последние изменения" }) # Spec-driven разработка Agent({ agent: "openspec-manager", prompt: "Примени change-спеку auth-refactor" })

Используй Forked Subagent когда:

  • Code review — «пересмотри наш выбор архитектуры» (pragmatist, conservative)
  • Критический взгляд — «найди проблемы в том, что мы сделали» (innovator)
  • Альтернативное решение — «предложи другой подход с учётом наших ограничений»
  • Контекст критичен — задача ссылается на предыдущие обсуждения
  • Дебаты — когда нужен взгляд «со стороны», но с полной информацией
# Типичные вызовы # Архитектурный критик Agent({ agent: "innovator", prompt: "Что не так с нашей текущей архитектурой?", fork: true }) # Прагматичный взгляд Agent({ agent: "pragmatist", prompt: "Оцени производительность нашего решения", fork: true }) # Консервативная проверка Agent({ agent: "conservative", prompt: "Есть ли риски в том, что мы планируем?", fork: true })

⚠️ Типичные ошибки

1. Забыть передать контекст в обычном субагенте

Проблема: Вызываешь обычный Agent и ожидаешь, что субагент знает о предыдущих решениях. Он не знает — его контекст чистый.

Решение: Либо передай всю необходимую информацию в prompt, либо используй fork: true.

2. Использовать fork для простых задач

Проблема: Форк субагента копирует всю историю. Если история на 100K токенов, ты тратишь 100K+ на копию, даже если задача — «найди функцию».

Решение: Fork — для задач, которые реально нуждаются в контексте. Для простых задач — обычный субагент.

3. Ожидать межсубагентной связи

Проблема: Думаешь, что два субагента могут общаться друг с другом. Не могут — каждый работает в своём изолированном контексте.

Решение: Если нужна связь — основной Claude координирует. Он получает результат от первого и передаёт второму.

4. Путать fork с наследованием

Проблема: Думаешь, что форкнутый субагент «продолжает» разговор. Он не продолжает — он получает мёртвую копию на момент вызова. Дальнейшие изменения в основном контексте не влияют на субагента, и наоборот.

Решение: Fork — это снимок, а не продолжение. После возврата субагента основной Claude работает с обновлённым контекстом.

5. Не учитывать лимиты токенов

Проблема: Forked Subagent получает копию контекста + свой собственный контекст. Суммарно может превысить лимит модели.

Решение: Следи за размером контекста. Если он близок к лимиту — обычный субагент безопаснее.

📖 Примеры из практики

Сценарий 1: Исследование без контекста

Задача: «Найди все эндпоинты, которые используют middleware аутентификации»

Решение: Обычный Subagent (explorer). Контекст не нужен — задача самодостаточна.

Agent({ agent: "explorer", prompt: "Найди все файлы в src/api/ где используется auth middleware. Верни список путей и строк." })

Сценарий 2: Критика решения

Задача: «Мы только что выбрали архитектуру очередей на Redis. Есть ли проблемы?»

Решение: Forked Subagent (innovator). Ему нужен контекст обсуждения.

Agent({ agent: "innovator", prompt: "Оцени наш выбор Redis Queue. Какие есть альтернативы и риски?", fork: true })

Сценарий 3: Документирование

Задача: «Напиши CHANGELOG под последние изменения»

Решение: Обычный Subagent (writer-agent). Ему нужно прочитать git diff, а не всю историю разговора.

Agent({ agent: "writer-agent", prompt: "Прочитай git diff HEAD~3 и допиши CHANGELOG.md" })

Итого: Оба типа субагентов — встроенные механизмы Claude Code для делегирования задач. Subagent — чистый контекст, минимум токенов, полная изоляция. Forked Subagent — копия разговора, нужен контекст, но дороже. Выбирай в зависимости от задачи: исследование — обычный, ревью — форкнутый.
