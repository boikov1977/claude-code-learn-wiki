🔌 Что такое MCP?

Знаешь USB? Раньше у каждого устройства был свой разъём — а потом появился USB как единый стандарт. Теперь любую периферию воткнул — и работает. MCP — это USB для AI-инструментов.

Проще простого: MCP (Model Context Protocol) — это протокол, через который Claude Code подключается к внешним сервисам и получает доступ к их инструментам. Claude может поискать в интернете, прочитать документацию, проверить что-то — всё это через MCP-серверы.

MCP-сервер — это программа, которая предоставляет Claude Code набор инструментов (tools). Claude подключается к серверу, видит какие инструменты есть, и может их вызывать.

Допустим, у тебя есть MCP-сервер для поиска в интернете. Claude хочет найти что-то → вызывает инструмент search_web на этом сервере → сервер ищет и возвращает результат. Claude даже не знает как ищет — он просто видит доступный инструмент.

Процесс:

  1. Запуск. При старте сессии Claude Code поднимает все настроенные MCP-серверы.
  2. Регистрация. Каждый сервер сообщает: «у меня есть инструменты A, B, C».
  3. Использование. Когда Claude решает, что ему нужен внешний инструмент — он вызывает его.
  4. Результат. Сервер выполняет операцию и возвращает данные Claude.
💡 Ключевой момент: Claude не программирует MCP-серверы на лету — они уже написаны и настроены заранее. Claude просто использует доступные инструменты.

🌐 Типы подключения MCP-серверов

MCP-серверы бывают трёх видов. Разница — в том, как Claude Code к ним подключается:

stdio Локальный процесс

Сервер запускается как процесс на твоей машине. Claude Code общается с ним через stdin/stdout — как с обычной программой.

# Пример: наш poisk-mcp { "type": "stdio", "command": "python", "args": ["server.py"] }

Плюс: быстрый, простой, не требует интернета.
Минус: занимает ресурсы твоей машины.

remote Удалённый сервер

Сервер живёт на удалённом URL. Claude Code подключается по HTTP, как браузер к сайту.

# Пример: context7 { "type": "remote", "url": "https://mcp.context7.com/mcp", "headers": { "API_KEY": "..." } }

Плюс: не нагружает машину, всегда актуален.
Минус: нужен интернет, нужен API-ключ.

sse Server-Sent Events

Разновидность удалённого подключения. Сервер отправляет события через Server-Sent Events — постоянный поток данных от сервера к клиенту. Используется реже, но подходит для долгоживущих соединений.

Тип Где живёт Пример Когда использовать
stdio На твоей машине poisk-mcp, openspec-mcp Локальные инструменты, быстрый доступ
remote На сервере провайдера context7 Онлайн-сервисы, обновляемые данные
sse На сервере провайдера Некоторые кастомные Долгие операции, стриминг

🌎 Глобальный MCP

Виден: основному агенту (мне) во всех проектах.

Где настраивается:

  • ~/.claude.json — новый формат (через claude mcp add)
  • ~/.claude/settings.json — старый формат (но тоже работает)

Пример: наш poisk-mcp — глобальный, потому что мне нужен поиск всегда.

🔒 Изолированный MCP

Виден: только конкретному агенту. Я про него не знаю, пока не вызову агента.

Где настраивается: в frontmatter агента, поле mcpServers.

Пример: openspec-mcp — только у openspec-manager. Поднимается только когда агент активен.

Пример изолированного MCP во frontmatter

--- # ~/.claude/agents/openspec-manager.md --- --- name: openspec-manager # ...остальные поля... mcpServers: openspec-mcp: type: stdio command: uvx args: - "openspec-mcp" --- # Описание агента...

Этот MCP-сервер только у openspec-manager. Никто другой его не видит.

⚠️ Ловушка: дублирование серверов

🚨 Наш реальный кейс с poisk-mcp

Была ситуация: poisk-mcp был прописан дважды:

  • В ~/.claude/settings.json — глобально
  • В ~/.claude.json — тоже глобально

В результате — сервер отваливался. Два экземпляра одного и того же сервера конфликтовали.

Решение: удалили из settings.json. Оставил только ~/.claude.json.

💡 Правило: Один сервер — одна запись. Если подключаешь глобально через claude mcp add — он попадёт в ~/.claude.json. Не дублируй в settings.json.

Где что живёт:

Файл Формат Когда использовать
~/.claude.json Новый (claude mcp add) Рекомендуется. Основной способ добавления MCP
~/.claude/settings.json Старый Можно, но лучше не смешивать с новым форматом
Frontmatter агента mcpServers Для изолированных серверов (только у этого агента)

🔒 Изоляция контекста: зачем это важно

Это самая мощная фича изолированных MCP-серверов.

Проблема: Если у тебя 10 MCP-серверов — основной агент видит все 100+ инструментов. Это засоряет контекст. Модель тратит токены на то, чтобы «знать» про инструменты, которыми она никогда не пользуется.

Решение: Изолированные MCP подключаются только к нужному агенту. Основной агент (я) их вообще не видит. Контекст чистый.

Пример:

Без изоляции (плохо)

У основного агента 15 инструментов от 5 MCP-серверов. Каждый вызов LLM тратит токены на описание всех 15 инструментов, даже если 12 из них не нужны.

С изоляцией (хорошо)

Основной агент видит только search_web от poisk-mcp. Остальные инструменты (context7, openspec-mcp) — знают только их агенты. Контекст минимален, токены экономятся.

Как это работает

Когда я (основной агент) запускаю субагента — тот поднимает свои MCP-серверы. Все инструменты субагента видны только ему. Я вижу только результат работы субагента, а не его внутренние инструменты.

# Я — основной агент Мои инструменты: Read, Write, Edit, Bash, search_web (poisk-mcp) # Вызываю innovator → innovator поднимает context7 → innovator видит: search_library_docs, get_library_version → Я вижу только: "Вот ответ от innovator: ..." # Вызываю openspec-manager → openspec-manager поднимает openspec-mcp → openspec-manager видит: create_change, apply_change, ... → Я вижу только: "Вот ответ от openspec-manager: ..."

Итого: MCP — это протокол подключения внешних инструментов к Claude Code. Серверы бывают stdio (локальные) и remote (онлайн). Глобальные видны основному агенту, изолированные — только конкретному субагенту. Не дублируй серверы в разных файлах. Изоляция экономит контекст и делает систему чище. 🔌
