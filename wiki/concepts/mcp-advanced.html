🔌 Транспорты: stdio, HTTP, SSE

MCP-серверы общаются с Claude Code через транспорт — способ передачи данных. Claude Code поддерживает три типа транспорта. Разница — в том, где живёт сервер и как к нему подключаются.

stdio Локальный процесс (stdin/stdout)

Как работает: Claude Code запускает процесс на твоей машине. Общение идёт через стандартные потоки ввода-вывода — Claude пишет JSON в stdin процесса, процесс отвечает в stdout. Никакого HTTP, никакой сети.

# Формат конфигурации { "type": "stdio", "command": "python", "args": ["server.py"], "env": { "API_KEY": "sk-..." } }
ХарактеристикаЗначение
СкоростьБыстрая — нет сетевого оверхеда
ЗависимостиТолько локальное окружение (Python, Node, Go...)
БезопасностьВысокая — данные не покидают машину
ОграничениеЗанимает ресурсы твоей машины (CPU, RAM)
Примерыpoisk-mcp, openspec-mcp, любые локальные серверы
💡 Стандартный выбор. Большинство MCP-серверов используют stdio. Это самый простой и надёжный транспорт — запустил и забыл. Используй stdio, пока нет причин делать по-другому.

http Удалённый сервер (JSON-RPC)

Как работает: Сервер живёт на удалённом URL. Claude Code отправляет JSON-RPC-запросы по HTTP(S), получает ответы. Тот же принцип, что и REST API, но с фиксированным форматом JSON-RPC 2.0.

# Формат конфигурации { "type": "http", "url": "https://mcp.example.com/mcp", "headers": { "Authorization": "Bearer sk-..." } }
ХарактеристикаЗначение
СкоростьЗависит от сети и латентности сервера
ЗависимостиТолько HTTP(S) — работает из любого окружения
БезопасностьЗависит от провайдера. Данные идут через сеть
ОграничениеНужен интернет и, как правило, API-ключ
Примерыcontext7, brave-search, github-mcp
⚠️ Новый стандарт. http заменяет устаревший sse. Если провайдер предоставляет оба варианта — выбирай http. Формат http поддерживает stateless-режим (без сессий) и stateful-режим (с session ID в заголовках).

sse Server-Sent Events (deprecated)

Как работает: Клиент открывает долгоживущее HTTP-соединение для получения событий от сервера (SSE), а запросы отправляет через отдельный POST-эндпоинт. Два канала: один для получения событий (GET), другой для отправки команд (POST).

# Формат конфигурации (устаревший) { "type": "sse", "url": "https://mcp.example.com/sse", "headers": { "API_KEY": "..." } }

🚨 Устаревший транспорт

sse объявлен deprecated в MCP-спеке. Новые серверы должны использовать http. Если провайдер всё ещё отдаёт только sse — подключай, но планируй миграцию. http проще, надёжнее и не требует двух отдельных эндпоинтов.

ХарактеристикаЗначение
СтатусDeprecated — не используй для новых серверов
ОсобенностьДва эндпоинта: GET для событий, POST для команд
ПроблемаСложнее отлаживать, проблемы с прокси и балансировщиками
Альтернативаhttp — один эндпоинт, stateless или stateful

📊 Сравнение транспортов

Транспорт Где живёт сервер Канал связи Статус Когда использовать
stdio На твоей машине stdin/stdout (JSON) ✅ Актуален Локальные инструменты, быстрый доступ, приватные данные
http На сервере провайдера HTTP(S) (JSON-RPC) ✅ Рекомендуется Онлайн-сервисы, обновляемые данные, SaaS-инструменты
sse На сервере провайдера SSE + POST ⚠️ Deprecated Только старые серверы без HTTP-поддержки

🏠 Scope: Local, User, Project

MCP-серверы можно подключить на трёх уровнях (scope). От зависит, кто видит сервер и где хранится конфигурация.

Local Уровень агента

Где живёт: Во frontmatter файла агента в ~/.claude/agents/

Кто видит: Только конкретный агент. Основной Claude Code про него не знает.

Когда использовать: Когда инструмент нужен только одному субагенту и не должен засорять контекст основного агента.

# ~/.claude/agents/openspec-manager.md --- name: openspec-manager model: inherit mcpServers: openspec-mcp: type: stdio command: uvx args: - "openspec-mcp" ---

Этот сервер виден только openspec-manager. Никто другой его не поднимает.

User Глобальный уровень

Где живёт: ~/.claude.json (в поле mcpServers)

Кто видит: Основной Claude Code во всех проектах.

Когда использовать: Инструменты, которые нужны всегда — поиск, базы знаний, универсальные утилиты.

# ~/.claude.json { "mcpServers": { "poisk-mcp": { "type": "stdio", "command": "/home/user/mcp/poisk-mcp/venv/bin/python", "args": ["/home/user/mcp/poisk-mcp/server.py"], "env": { "HF_HUB_OFFLINE": "1" } } } }

Добавляется командой claude mcp add (см. раздел CLI).

Project Проектный уровень

Где живёт: .mcp.json в корне проекта

Кто видит: Все Claude Code-сессии в этом проекте.

Когда использовать: Серверы, специфичные для проекта — документация проекта, внутренние API, тестовые базы данных.

# .mcp.json в корне проекта { "mcpServers": { "project-docs": { "type": "stdio", "command": "npx", "args": ["@my-org/project-docs-mcp"] } } }

Команда: claude mcp add -s project ...

⚠️ Безопасность: Файл .mcp.json может содержать токены и ключи. Если проект публичный — добавь его в .gitignore. Используй переменные окружения (см. раздел «Переменные окружения»).

🎯 Приоритеты scope

Когда MCP-сервер с одним именем определён на нескольких уровнях — побеждает более узкий scope:

Local #1
>
Project #2
>
User #3

Пример конфликта

Допустим, сервер search определён в трёх местах:

# 1. User (~/.claude.json) — глобальный { "search": { "type": "stdio", "command": "search-v1" } } # 2. Project (.mcp.json) — проектный { "search": { "type": "stdio", "command": "search-v2" } } # 3. Agent (frontmatter) — локальный агента { "search": { "type": "stdio", "command": "search-v3" } }

Для основного агента — используется search-v2 (Project > User).

Для агента с Local-определением — используется search-v3 (Local > всё).

Приоритет Local > Project > User — более узкий scope перекрывает широкий.

💻 Управление через Claude CLI

Claude Code предоставляет команды для управления MCP-серверами из терминала. Все команды работают через claude mcp:

claude mcp add — добавить сервер

# Добавить stdio-сервер (по умолчанию — user scope) claude mcp add search-server \ -s user \ -- python /path/to/server.py # Добавить с переменными окружения claude mcp add my-server \ -s user \ -e API_KEY=sk-xxx \ -e DEBUG=true \ -- node /path/to/server.js # Добавить HTTP-сервер claude mcp add context7 \ -s user \ --transport http \ --url https://mcp.context7.com/mcp # Добавить в проект (.mcp.json) claude mcp add project-tool \ -s project \ -- npx @my-org/project-mcp

Флаги:

ФлагОписание
-s, --scopeuser (по умолчанию), project
-e, --envПеременная окружения (можно указать несколько)
--transportstdio (по умолчанию), http, sse
--urlURL для HTTP/SSE транспорта

claude mcp list — список серверов

# Показать все настроенные MCP-серверы claude mcp list # Вывод: # poisk-mcp (stdio) — /home/user/mcp/poisk-mcp/server.py # context7 (http) — https://mcp.context7.com/mcp

Показывает все серверы со всеми scope: user, project, local (в frontmatter агентов).

claude mcp get — посмотреть конфиг сервера

# Посмотреть конфиг конкретного сервера claude mcp get poisk-mcp # Вывод покажет type, command, args, env, scope

Полезно для отладки — убедиться, что сервер подключён правильно.

claude mcp remove — удалить сервер

# Удалить сервер по имени claude mcp remove poisk-mcp # Удалить из конкретного scope claude mcp remove poisk-mcp -s project

Удаляет запись из ~/.claude.json (user scope) или .mcp.json (project scope). Из ~/.claude/settings.json удаляется вручную (старый формат).

claude mcp reset — сбросить индекс

# Сбросить кэш MCP-серверов (полезно при багах) claude mcp reset

Сбрасывает внутренний кэш инструментов. Используй, если Claude Code «не видит» инструменты от подключённого сервера.

🔒 Пермишены для MCP-инструментов

MCP-инструменты подчиняются системе пермишенов Claude Code. Каждый инструмент идентифицируется по формату mcp__{server}__{tool}, и его можно разрешить глобально или вручную.

Формат именования

# Формат: mcp____ # Примеры: mcp__poisk-mcp__search_web mcp__poisk-mcp__read_webpage mcp__context7__resolve-library-id mcp__context7__get-library-docs mcp__openspec-mcp__create_change

Два подчёркивания __ — разделитель между namespace-ами. Первый — имя MCP-сервера, второй — имя инструмента на этом сервере.

Автоматическое разрешение с wildcard-ами

Вместо того чтобы разрешать каждый инструмент по отдельности, можно использовать паттерны с *:

# ~/.claude/settings.local.json { "permissions": { "allow": [ # Разрешить все инструменты от poisk-mcp "mcp__poisk-mcp__*", # Разрешить конкретный инструмент от context7 "mcp__context7__get-library-docs", # Разрешить все инструменты от всех MCP-серверов "mcp__*__*" ] } }

Паттерны:

ПаттернЧто разрешает
mcp__poisk-mcp__*Все инструменты от poisk-mcp
mcp__*__*Все инструменты от всех MCP-серверов
mcp__context7__get-library-docsТолько конкретный инструмент
💡 Подсказка: Когда Claude Code впервые вызывает MCP-инструмент, он покажет запрос разрешения. Если нажать «Allow always» — правило добавится в settings.local.json автоматически. Но ручная настройка надёжнее.

Полный пример из нашего конфига

# ~/.claude/settings.local.json { "permissions": { "allow": [ "mcp__poisk-mcp__search_web", # поиск "mcp__poisk-mcp__read_webpage", # чтение страниц "Bash(claude mcp *)", # управление MCP "Bash(env)", # переменные окружения "WebSearch" # встроенный поиск ] } }

🌡️ Переменные окружения в конфигах

Передавать API-ключи и секреты напрямую в JSON — плохая идея. Claude Code поддерживает подстановку переменных окружения:

✅ Правильно: через env

{ "type": "stdio", "command": "node", "args": ["server.js"], "env": { "API_KEY": "${MY_API_KEY}" } }

Ключ берётся из переменной окружения $MY_API_KEY. Не хранится в файле, не попадёт в git.

❌ Неправильно: хардкод

{ "type": "stdio", "command": "node", "args": ["server.js"], "env": { "API_KEY": "sk-live-abc123secretkey" } }

Токен в открытом виде в файле. Если проект в git — токен утек.

Где задавать переменные

СпособДля какого scope
~/.bashrc / ~/.zshrcВсе scope — переменная доступна везде
claude mcp add -e KEY=valUser scope — задаётся при добавлении
.env файл + exportПроектный scope — только в этом проекте
Блок env в конфиге MCPТолько для конкретного сервера

⭐ Best Practices

✅ Делай

  • Используй stdio для локальных серверов — проще и быстрее
  • Храни токены в переменных окружения, не в JSON
  • Используй claude mcp add вместо ручного редактирования .claude.json
  • Разделяй серверы по scope: глобальные в user, проектные — в project
  • Изолируй MCP-серверы субагентов через frontmatter
  • Добавляй .mcp.json в .gitignore если проект публичный
  • Используй wildcard mcp__server__* для автоматического разрешения
  • Проверяй конфиг через claude mcp get <name> после добавления

❌ Не делай

  • Не хардкодь API-ключи в .mcp.json или .claude.json
  • Не дублируй один сервер в settings.json и .claude.json
  • Не используй sse для новых серверов — только http
  • Не подключай все MCP-серверы глобально — засоряешь контекст
  • Не используй mcp__*__* если не доверяешь всем серверам
  • Не коммить .mcp.json с секретами
  • Не используй старый ~/.claude/settings.json для MCP — лучше .claude.json
  • Не запускай MCP-серверы без || true в хуках — могут заблокировать запись

📋 Чек-лист перед коммитом

  1. Проверь, что .mcp.json в .gitignore (если содержит секреты)
  2. Проверь, что токены заданы через переменные окружения
  3. Проверь, что нет дублей серверов в разных scope
  4. Проверь командой claude mcp list — всё на месте?
  5. Если проект团队ный — убедись, что .mcp.json содержит только публичные серверы

⚠️ Типичные ошибки

1. Токены в .mcp.json

Симптом: API-ключи в открытом виде в файле .mcp.json, который попал в git.

Решение:

# 1. Убери токен из .mcp.json # 2. Перенеси его в переменную окружения export MY_API_KEY="sk-xxx" # 3. Обнови .mcp.json — используй подстановку { "env": { "API_KEY": "${MY_API_KEY}" } } # 4. Добавь в .gitignore echo ".mcp.json" >> .gitignore # 5. Если токен уже в истории git — rotated!

2. Авто-approve всех инструментов

Симптом: Используешь mcp__*__* и позволяешь всем MCP-серверам всё без подтверждения.

Проблема: Если подключён новый MCP-сервер (даже временно) — он автоматически получает полный доступ. Небезопасно.

Решение: Разрешай серверы точечно:

# Плохо: "mcp__*__*" # Хорошо: "mcp__poisk-mcp__*" "mcp__context7__*" "mcp__openspec-mcp__create_change"

3. Дублирование серверов

Симптом: Один и тот же сервер прописан и в ~/.claude.json, и в ~/.claude/settings.json. Сервер отваливается или ведёт себя странно.

Решение: Используй только один источник. Рекомендуется ~/.claude.json через claude mcp add.

# Проверь, нет ли дубля: cat ~/.claude.json | python3 -c "import sys,json; print(json.dumps(json.load(sys.stdin).get('mcpServers',{}), indent=2))" # Если сервер есть и там, и в settings.json — удали из settings.json

4. MCP-сервер не виден

Симптом: claude mcp list показывает сервер, но Claude Code не видит его инструменты.

Возможные причины:

  • Сервер упал при запуске (проверь stderr)
  • Неверный путь к исполняемому файлу
  • Не хватает зависимостей (Python-пакеты, Node-модули)
  • Проблема с сетью (для HTTP/SSE серверов)
  • Кэш — попробуй claude mcp reset

5. Переменные окружения не подставляются

Симптом: В конфиге MCP написано "${MY_KEY}", но сервер получает пустую строку.

Причина: Переменная задана в .bashrc, но Claude Code запущена из другого терминала / через systemd / через IDE.

Решение: Проверь через env | grep MY_KEY. Если переменной нет — задай в ~/.profile или через claude mcp add -e MY_KEY=value.

6. SSE вместо HTTP

Симптом: Провайдер предоставляет оба варианта, а ты подключил sse.

Почему это плохо: sse deprecated, сложнее отлаживать, проблемы с прокси и балансировщиками, два эндпоинта вместо одного.

Решение: Переключись на http:

# Удали старый claude mcp remove my-server # Добавь заново с http claude mcp add my-server \ -s user \ --transport http \ --url https://example.com/mcp

Итого: MCP в Claude Code — это три транспорта (stdio http sse), три scope (Local User Project) с приоритетом узкого над широким. Управляй серверами через claude mcp add/list/get/remove. Разрешай инструменты через mcp__server__tool. Храни токены в переменных окружения, не дублируй серверы.
