🧠 Что такое память в Claude Code?

Забудь всё, что знаешь про «память» в человеческом смысле. Модель ничего не помнит между сессиями. Каждый раз когда ты запускаешь Claude Code — это чистый лист.

Проще простого: Память Claude Code — это текстовые markdown-файлы, которые загружаются в контекст при старте сессии. Никакой магии. Просто файлы. Claude читает их и «знает» то, что ты туда записал.

Представь: каждый раз при запуске Claude Code «открывает книгу» и читает несколько глав. Вот эти главы — и есть твоя память. Где они лежат и кто их читает — разбираемся дальше.

Три уровня, три типа файлов:

Уровень Где лежит Что содержит Когда читается

🌍 Глобальная память — ~/CLAUDE.md

Единственный файл. Лежит в домашней папке. Загружается в каждой сессии, независимо от того, в каком проекте ты работаешь.

# Структура ~/CLAUDE.md (у нас) # Глобальные инструкции для Андрея ## О пользователе - Меня зовут Андрей - Русскоязычный, неформальный стиль - Экспериментатор, прямолинейный ## Инструменты - Поиск: только poisk-mcp (встроенный WebSearch не работает) - Модель: claude-go-deepseek-v4-flash ## Агенты — ~/.claude/agents/ - openspec-manager — spec-driven разработка - writer-agent — документация - explorer — быстрый поиск по коду - ...ещё несколько ## Глобальная конфигурация - Хуки: бэкапы + автоформатирование - MCP: poisk-mcp - Пермишены: auto
Зачем глобальная?
Чтобы не повторять одно и то же в каждом проекте. Профиль, настройки инструментов, список агентов — это общее для всех проектов. Написал один раз в ~/CLAUDE.md — и Claude знает тебя везде.
💡 Главное правило: Держи ~/CLAUDE.md коротким — лучше меньше 50 строк. Claude читает его каждый раз, и длинный файл жрёт контекст.

📁 Проектная память

Живёт в корне проекта и загружается только когда ты работаешь именно в нём.

./CLAUDE.md

  • Инструкции специфичные для проекта
  • Как запускать, как тестировать
  • Архитектурные решения
  • Ты пишешь сам или через /init

.claude/ директория

  • settings.json — пермишены, хуки
  • agents/ — кастомные агенты проекта
  • memory/MEMORY.md — auto-memory
  • Часть настроек автоматическая

🤖 Auto-memory — Claude запоминает сам

Claude Code умеет сам записывать в MEMORY.md то, что узнал о проекте в процессе работы. Ты не просишь — он сам решает что стоит запомнить.

Как это работает

  1. Claude работает с проектом и замечает что-то важное
  2. Пишет запись в .claude/projects/*/memory/MEMORY.md
  3. При следующем старте — загружает первые 200 строк этого файла
  4. Получается: Claude «учится» на предыдущих сессиях
# Пример записей в MEMORY.md (auto-generated) ## Архитектура - Проект использует FastAPI + PostgreSQL - Миграции через Alembic - Тесты на pytest ## Ключевые файлы - `app/main.py` — точка входа - `app/models.py` — SQLAlchemy модели - `tests/conftest.py` — фикстуры
⚠️ Важно: Auto-memory — это не волшебная телепатия. Claude записывает в файл то, что кажется ему важным. Иногда он угадывает, иногда — нет. Всегда можно подредактировать MEMORY.md вручную.

📄 Подгрузка по запросу — @-ссылки

Самый простой способ дать Claude доступ к любому файлу — написать @путь/к/файлу прямо в чате. Файл прочитается и его содержимое попадёт в контекст.

Примеры

# Просто файл @src/utils/helpers.py # С папкой @docs/api-spec.md # С-relative путём ../shared/config.yaml # Или прямо в сообщении Прочитай @config.json и объясни структуру

Ключевое отличие от остальных уровней: @-ссылки не постоянные. Они подгружают файл один раз — для текущего сообщения. В следующем сообщении этого файла уже не будет, если снова не указать.

Пошагово:

1. Глобальный контекст Читается ~/CLAUDE.md — профиль, настройки, инструменты. Это происходит всегда.
2. Определение проекта Claude проверяет, в какой директории ты находишься. Если это проект — загружается его контекст.
3. Проектный контекст Читаются ./CLAUDE.md и .claude/settings.json. Эти файлы специфичны для проекта.
4. Auto-memory Загружаются первые 200 строк MEMORY.md — то, что Claude запомнил о проекте.
5. Готово к работе Claude имеет полный контекст и готов отвечать. @-ссылки подгружаются по необходимости позже.
💡 Контекст — это ресурс. Каждый прочитанный файл занимает место в контекстном окне. Чем короче ~/CLAUDE.md и MEMORY.md — тем больше места для реальной работы.

🔗 Связи между записями

В MEMORY.md можно ссылаться на другие записки двойными скобками: [[имя-записи]]. Это просто маркер связи — работает как ссылка на другую запись.

# Пример связей в MEMORY.md ## Архитектура - Система аутентификации описана в [[auth-setup]] - Деплой настраивался в [[deploy-notes]] ## Решения - Выбрали PostgreSQL вместо MySQL — см. [[db-choice]] - API версионирование через URL: [[api-versioning]]

Ссылки могут вести на записи, которые ещё не созданы. Это просто маркеры — потом можно будет добавить содержимое по этим именам.

🛠️ Как мы настроили

Наша стратегия простая: всё общее — в глобал, ничего лишнего — в проекты.

Что мы сделали

  1. Написали ~/CLAUDE.md — профиль, инструкции по поиску, список агентов, модель
  2. Удалили проектные MEMORY.md — старые файлы вроде -home-andrei-... больше не нужны
  3. Весь общий контекст уехал глобально — чтобы не дублировать в каждом проекте

✅ До

  • Каждый проект имел свою MEMORY.md
  • Дублирование настроек между проектами
  • Старые файлы вида -home-andrei-...
  • Путаница: где что лежит?

🎯 После

  • Один ~/CLAUDE.md на все проекты
  • Проектные файлы чистые и по делу
  • Нет мусора
  • Понятная структура

🎯 Практические советы

1. Держи ~/CLAUDE.md коротким

Меньше 50 строк. Каждая строка жрёт контекст. Если информация нужна редко — подгружай через @, а не держи в глобальном файле.

2. Не дублируй то, что уже есть в коде

Если структура проекта видна из файловой системы — не пиши про неё в MEMORY.md. Claude сам разберётся. Записывай только то, что нельзя узнать из кода: архитектурные решения, контекст, историю выбора.

3. Используй даты для ротации

Когда обновляешь запись — ставь дату. Устаревшее можно удалить или архивировать.

# Хорошо — видно когда обновлено ## Настройка деплоя (обновлено 2025-06-10) - Используем Docker Compose - Порт: 8080 # Плохо — непонятно актуально ли ## Настройка деплоя - Используем Docker Compose

4. Односложные записи лучше длинных

Claude не нуждается в красивых предложениях. Короткие, ёмкие записи работают лучше:

# Плохо — много воды В проекте используется база данных PostgreSQL, которая была выбрана после долгого обсуждения различных вариантов, включая MySQL и SQLite... # Хорошо — по делу БД: PostgreSQL. Выбор — см. [[db-choice]]

5. Проверяй auto-memory

Claude записывает не всегда то, что ты хочешь. Периодически заглядывай в MEMORY.md и правь: убирай лишнее, добавляй важное, обновляй устаревшее.

⚡ Лайфхак: Если Claude «забыл» что-то важное — скорее всего ты просто не написал это в ~/CLAUDE.md или MEMORY.md. Он не угадывает — он читает файлы. Запиши — и запомнит.

Итого: Память Claude Code — это markdown-файлы, которые читаются при старте. ~/CLAUDE.md — глобально, ./CLAUDE.md — для проекта, @ — по запросу. Claude сам запоминает через auto-memory. Простая система, если знаешь где копать. 🧠
