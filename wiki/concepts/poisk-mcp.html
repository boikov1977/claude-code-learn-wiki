🔍 Что такое poisk-mcp?

Claude Code умеет читать файлы, запускать команды и общаться с MCP-серверами. Но вот беда — встроенный инструмент WebSearch не работает через OpenCode-прокси (DeepSeek выдаёт ошибку схемы). А часто нужно погуглить что-то прямо из сессии: документацию, ошибку, свежую новость.

poisk-mcp — это наш собственный MCP-сервер для поиска в интернете. Мы написали его с нуля на Python и подключили к Claude Code через stdio. Он ищет сразу в двух поисковых системах (DuckDuckGo и SearXNG), отсеивает дубли, ранжирует результаты нейросетью по смысловой релевантности и разнообразит выдачу.

Проще простого: poisk-mcp — это как подключить к Claude свой персональный Google, только бесплатный, с умной сортировкой и без API-ключей. Запрос уходит — ответ приходит. Как обычный поиск, только внутри Claude Code.

Ключевые модули:

  • server.py — точка входа, объявляет MCP-инструменты, связывает модули
  • engine.py — ядро: мультиязычный поиск, агрегация из бэкендов, дедупликация, ранжирование, DiversityEngine
  • backends.py — поисковые бэкенды (DuckDuckGo, SearXNG)
  • reranker.py — нейро-ранжирование (FlashRank cross-encoder)
  • network.py — безопасные HTTP-запросы с SSRF-защитой
  • cache.py — кэш результатов и rate limiter
  • models.py — data-классы SearchResult, ProcessedQuery

🧰 Инструменты

poisk-mcp предоставляет Claude четыре инструмента. Вот что умеет каждый:

Инструмент Назначение Параметры
search_web Поиск в интернете с нейро-ранжированием. Основной инструмент. query (строка), max_results (8, до 20), backend ("auto"), rerank (true), diversity (true), languages ("" — одноязычный, "ru,en" — мультиязычный)

web_research Глубокое исследование темы. Ищет, открывает топ-страницы, анализирует и генерирует отчёт. query (строка), depth (1–5, по умолчанию 3)

Проходит по шагам:

  1. Claude вызывает инструмент через stdio (JSON-RPC)
  2. Кэш проверяется — если тот же запрос был менее часа назад, результат отдаётся мгновенно
  3. Rate limiter проверяет, не превышен ли лимит в 30 запросов в минуту
  4. Мультиязычный поиск — если указан параметр languages, запрос размножается по регионам (например, ru-ru, us-en, cn-zh)
  5. Бэкенды выполняют поиск параллельно: DuckDuckGo (для каждого языка) + SearXNG (через Docker)
  6. Дедупликация — если ссылка встречалась в нескольких бэкендах или регионах, она попадёт в список один раз
  7. Нейро-ранжирование — FlashRank cross-encoder сравнивает запрос с каждым результатом и оценивает релевантность
  8. Jaccard MMR-разнообразие — чтобы в выдаче не было 5 похожих результатов, алгоритм чередует по смыслу и доменам
  9. Результат возвращается Claude в виде структурированного текста с URL, описаниями и оценками релевантности

🌐 Бэкенды

🦆 DuckDuckGo

Основной бэкенд. Не требует API-ключей, работает из коробки.

  • Использует библиотеку ddgs (или duckduckgo_search)
  • Поддерживает мультиязычный поиск: регионы выбираются из параметра languages
  • В одноязычном режиме использует регион по умолчанию
  • Также умеет искать новости и изображения

🌐 SearXNG

Мета-поисковик с агрегацией из 70+ поисковых систем.

  • Локальный Docker-инстанс на localhost:8081
  • Публичные инстансы как fallback: searx.be, search.disroot.org и другие
  • Авто-переключение: если один инстанс упал — пробует следующий
  • Работает через REST API с JSON-ответами

Порядок работы: В режиме auto (по умолчанию) бэкенды опрашиваются параллельно. DuckDuckGo (со всеми указанными регионами) и SearXNG работают одновременно. Результаты объединяются, дедуплицируются, ранжируются FlashRank и проходят через Jaccard MMR-диверсификацию.

🌍 Мультиязычный поиск

Главная фишка v3.5 — мультиязычный поиск. Параметр languages в search_web позволяет искать по одному запросу сразу в нескольких языковых регионах.

# Искать на русском, английском и китайском search_web("последние новости ИИ", languages="ru,en,zh") # Только английский search_web("latest AI news", languages="en") # Обычный поиск (без мультиязычности) search_web("новости")

Поддерживаемые языки: ru, en, zh, de, fr, es, ja, ko

Как работает

  1. Запрос отправляется в DuckDuckGo для каждого языка со своим регионом (ru-ru, us-en, cn-zh и т.д.)
  2. Результаты со всех регионов собираются в общий пул
  3. Дедупликация по URL — один и тот же сайт не дублируется
  4. FlashRank переранжирует общий пул по релевантности запросу
  5. DiversityEngine выбирает максимально разные по смыслу результаты

Когда использовать

Мультиязычный поиск полезен для:
• Поиска технической документации (en + zh + ru)
• Новостей о технологиях (en + de + ja)
• Сравнения подходов в разных языковых сообществах
• Поиска научных статей (en + zh + de)

🤖 Нейро-ранжирование (FlashRank)

Обычные поисковики сортируют по ключевым словам и ссылочному весу. А мы делаем кое-что поинтереснее — прогоняем результаты через cross-encoder, который понимает смысл.

FlashRank cross-encoder

Используется модель FlashRank — это ONNX-модель (~98MB), которая не требует PyTorch. Она работает на любом CPU через ONNX Runtime.

Принципиальное отличие от bi-encoder:

  • Bi-encoder (как было раньше): вычисляет эмбеддинги для запроса и каждого результата по отдельности, затем считает косинусное сходство
  • Cross-encoder (FlashRank): подаёт пару (запрос, результат) на вход одной модели и получает прямую оценку релевантности

Cross-encoder работает точнее, потому что модель видит оба текста одновременно и может улавливать тонкие смысловые связи, которые bi-encoder пропускает.

Без PyTorch

Раньше для нейро-ранжирования использовался PyTorch с sentence-transformers (~2.5GB). FlashRank работает через ONNX Runtime — лёгкий рантайм без CUDA. Вся установка: pip install -r requirements.txt за 30 секунд. Модель скачивается автоматически при первом запуске.

🎯 Диверсификация результатов (MMR)

Даже после ранжирования может получиться так, что первые 5 результатов — с одного сайта или очень похожи по смыслу. Алгоритм MMR (Maximal Marginal Relevance) решает эту проблему.

Jaccard MMR

В v3.5 DiversityEngine использует Jaccard-based MMR вместо нейросетевого:

  • Сравнивает тексты результатов через Jaccard-расстояние (пересечение слов)
  • Не требует эмбеддингов — работает всегда, без fallback
  • Параметр diversity=True (по умолчанию включён)

Domain MMR

Дополнительный режим: если нужно гарантированно получить ссылки с разных доменов, DiversityEngine чередует результаты по доменам, сохраняя релевантность.

Раньше было иначе

В предыдущих версиях Diversity требовал эмбеддингов от sentence-transformers. Если нейросеть была недоступна — использовалась упрощённая доменная стратегия. В v3.5 Jaccard MMR работает всегда.

🛡️ Защита и безопасность

MCP-сервер работает с сетью — значит, нужно думать о безопасности. Вот что реализовано в poisk-mcp:

Мера Что защищает Как реализовано
SSRF-защита От запросов к внутренним серверам (127.0.0.1, 10.x.x.x, 192.168.x.x и т.д.) Функция safe_ip() проверяет каждый IP перед запросом. Блокирует private, loopback, reserved, multicast, link-local адреса и метаданные облачных провайдеров (169.254.169.254)
Rate Limiter От спам-запросов и блокировок поисковиков Token bucket на 30 RPM (30 запросов в минуту). Если лимит исчерпан — запрос ждёт или отклоняется
TTL-кэш От повторных запросов к поисковикам LRU-кэш на 1000 записей с TTL 1 час. Один и тот же запрос не будет выполняться повторно в течение часа
DNS-кэш От DNS-спуфинга и частых резолвов Результат DNS-резолва кэшируется на 5 минут. Каждый резолв проверяется на SSRF
Ограничение редиректов От открытых петель редиректов и SSRF через редиректы Максимум 5 редиректов. Каждый промежуточный URL проверяется через SSRF-фильтр. Петли отслеживаются
Ограничение размера От переполнения памяти при чтении страниц Максимум 3MB на скачивание, 15000 символов на вывод. Страницы читаются чанками

Зачем это нужно?

Представь: Claude запрашивает read_webpage("http://169.254.169.254/latest/meta-data/") — это попытка прочитать метаданные облачного сервера (AWS, GCP). Без SSRF-защиты это сработало бы и утечёк бы секреты. С защитой — запрос блокируется на этапе DNS-резолва. Аналогично с запросами к localhost:xxxx — poisk-mcp не даст прочитать то, что слушает на внутреннем порту.

🔌 Как подключить

Три способа — от самого простого до самого гибкого:

1. Через CLI (рекомендуется)

# Добавить poisk-mcp глобально claude mcp add poisk-mcp --stdio \ /home/andrei/mcp/poisk-mcp/venv/bin/python \ /home/andrei/mcp/poisk-mcp/server.py # Проверить, что сервер подключился claude mcp list

2. Через ~/.claude.json

Добавь запись в секцию mcpServers:

{ "mcpServers": { "poisk-mcp": { "command": "/home/andrei/mcp/poisk-mcp/venv/bin/python", "args": ["/home/andrei/mcp/poisk-mcp/server.py"], "env": {} } } }

3. Как изолированный MCP у субагента

Если нужно, чтобы только конкретный субагент имел доступ к поиску (а основной Claude — нет), используй изолированный MCP во frontmatter:

--- name: my-researcher model: inherit mcpServers: poisk-mcp: command: /home/andrei/mcp/poisk-mcp/venv/bin/python args: - /home/andrei/mcp/poisk-mcp/server.py --- Ты — исследователь. Используй search_web для поиска информации.
💡 Совет: Для работы poisk-mcp нужен Python 3.10+ и виртуальное окружение с установленными зависимостями. Скрипт deploy.sh из репозитория создаёт venv, ставит все пакеты одной командой. Модель FlashRank (~98MB) скачивается автоматически при первом запуске сервера.

📋 Что изменилось в v3.5

Создано writer-agent на основе документации poisk-mcp.
