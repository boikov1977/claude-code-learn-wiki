Многоагентный Code Review

Самодельный рецепт — Workflow tool + Agent tool + методики review. Не встроенная фича Claude Code.

Это не встроенная фича Claude Code. Это самодельный рецепт — комбинация Workflow tool, Agent tool и методик code review. Термин «fan-out» пришёл из параллельного программирования: задача раздаётся N независимым workers, результаты собираются. В Claude Code такую оркестрацию пишут скриптом через parallel() в Workflow tool. Встроенный скилл code-review (вызывается через /code-review) работает иначе — он одноагентный и не поддерживает параллельную проверку с разными фокусами.

💡 Зачем многоагентный ревью

Один агент смотрит код — видит то, на что настроен его промпт. Три агента с разными фокусами (баги, безопасность, качество) видят больше. Это принцип four-eyes principle в исполнении Claude Code.

Практика code review показывает: один reviewer находит ~60% проблем, два независимых — ~75%, три — ~85%. С агентами то же самое: разные фокусы = меньше слепых зон.

Почему именно независимые

Не три одинаковых агента, а три разных. Если все ищут одни и те же паттерны — benefit от параллелизма равен нулю. Секрет в разных фокусах: один смотрит на баги, другой — на безопасность, третий — на качество кода. У каждого свой промпт, своя перспектива, свой «слепой угол», который закрывают другие.

🏗️ Архитектура: три фазы

Многоагентный code review строится из трёх последовательных фаз. Каждая фаза делает своё дело и передаёт результаты следующей.

Фаза 1: Найти (Find) — параллельно

Три агента запускаются параллельно через parallel() в Workflow tool. Каждый получает один и тот же diff (или файлы), но работает в своей роли:

  • Bug Hunter (model: inherit) — ищет логические ошибки, гонки, утечки, edge cases, null-deref
  • Security Auditor (model: inherit) — ищет injection, hardcoded secrets, XSS, insecure defaults
  • Code Quality (model: inherit) — ищет code smells, нарушения конвенций, дублирование, сложность

На выходе — три списка findings (находок). Каждая finding имеет серьёзность, описание, место в коде и обоснование.

Фаза 2: Проверить (Verify) — adversarial

Это критичный шаг, который большинство пропускает. Каждый finding из фазы 1 отправляется независимому скептику — отдельный агент с adversarial промптом. Скептик проверяет:

  • Правда ли это проблема, или false positive?
  • Какова реальная эксплуатируемость?
  • Насколько критична — P0 или P3?

Находки, которые скептик не подтверждает, отфильтровываются. Это снижает шум и экономит время разработчику.

Фаза 3: Собрать (Synthesize)

Все подтверждённые findings собираются в единый отчёт. Агент-синтезатор группирует по приоритету, убирает дубли, добавляет рекомендации по исправлению. На выходе — структурированный отчёт, готовый к использованию.

🐛 Пример: типичные уязвимости в Python-проекте

Допустим, у нас есть Python-проект с API и базой данных. Вот фрагмент кода, который будут проверять три агента.

Уязвимый код

# config.py import os API_KEY = "sk-EXAMPLE1234567890abcdef" # ← hardcoded secret DEBUG = True # db.py import sqlite3 def search_users(query: str): conn = sqlite3.connect("app.db") cursor = conn.cursor() # SQL injection — f-string в запросе cursor.execute(f"SELECT * FROM users WHERE name LIKE '%{query}%'") results = cursor.fetchall() return results # routes.py (FastAPI) from fastapi import FastAPI, Request from starlette.responses import JSONResponse import traceback router = FastAPI() @router.get("/search") def search(q: str = ""): results = search_users(q) return JSONResponse(results) @router.get("/users/{user_id}") def get_user(user_id: int): try: user = db.get_user(user_id) return JSONResponse(user) except Exception as e: # Утечка traceback клиенту return JSONResponse({"error": traceback.format_exc()}, status_code=500)

Что нашёл каждый агент

Bug Hunter

  • Finding #1: SQL Injection в search_users. Формируется raw SQL через f-string. Любой пользовательский ввод попадает напрямую в запрос — можно извлечь всё содержимое базы.
  • Finding #2: Нет валидации параметра q на входе в search. Если пустая строка — LIKE '%%' вернёт все записи, потенциально thousands of rows.

Security Auditor

  • Finding #3: Hardcoded API key в config.py. Доступен в git-истории даже после удаления файла — нужна ротация ключа.
  • Finding #4: Утечка отладочной информации в get_user. traceback.format_exc() возвращает полный стек-трейс клиенту, включая пути файлов и фрагменты кода.

Code Quality

  • Finding #5: Нет обработки ошибок при подключении к SQLite — если файл базы не найден, приложение упадёт с необработанным исключением.
  • Finding #6: Соединение с базой не закрывается в search_users — утечка соединений при многократных вызовах.
  • Finding #7: Нет type hints и docstrings у функций в db.py.

Что сделал скептик

Finding Вердикт скептика Действие
#1 SQL Injection Подтверждено — критично, эксплуатируемо через параметр q P0 — заменить на параметризованный запрос
#2 Нет валидации Частично — зависит от бизнес-логики, но SQL injection покрывает эту уязвимость P2 — добавить валидацию, ограничить длину
#3 Hardcoded secret Подтверждено — staging или production? P0 — вынести в переменные окружения, ротировать ключ
#4 Утечка отладки Подтверждено — полный стек-трейс клиенту раскрывает внутреннюю структуру приложения P1 — логировать ошибки на сервере, клиенту отдавать общее сообщение
#5 Нет обработки ошибок DB Отклонено — SQLite.raises если файл не найден, но это ожидаемое поведение при старте False positive
#6 Утечка соединений Подтверждено — реальная проблема при высокой нагрузке P1 — использовать контекстный менеджер или connection pool
#7 Нет type hints Отклонено — это не баг, это стиль False positive

Финальный отчёт

# Итоговый отчёт — 4 из 7 findings прошли проверку P0 (критично): 1. SQL Injection в search_users — raw SQL через f-string, нужен parameterized query 2. Hardcoded API key в config.py — вынести в env-переменные P1 (важно): 3. Утечка traceback клиенту — логировать на сервере, отдавать generic error 4. Утечка соединений с SQLite — контекстный менеджер или pool P2 (средний): 5. Нет валидации параметра q — ограничить длину, sanitize # Отброшено: 2 false positives

⭐ Best Practices

1. Разные фокусы — не запускай трёх одинаковых агентов

Если все ищут одни и те же паттерны — параллелизм бесполезен. Один — баги, другой — безопасность, третий — качество. У каждого свой промпт, свой список паттернов, своя модель проверки.

2. Diff + контекст — давай не только diff

Diff показывает что изменилось, но не показывает как это используется. Передавайте агентам не только diff, но и: маршруты API, модели данных, конфигурацию, middleware. Это снижает false positives и повышает точность.

3. Verify обязателен — иначе 50% findings будут false positives

Не верьте агентам на слово. Каждая находка должна пройти проверку скептиком (adversarial verify). Без этого вы тонете в false positives и разработчики перестают доверять ревью.

4. Писать schema для structured output

Если агенты отвечают свободным текстом — вы не можете автоматизировать обработку. Задавайте JSON schema для каждого типа агента:

// Схема для Bug Hunter { "findings": [ { "severity": "P0|P1|P2|P3", "category": "logic|security|performance|style", "file": "string", "line": "number", "description": "string", "confidence": "number (0-1)", "suggestion": "string" } ] }

Пример реального вывода агента после обработки кода:

// Пример вывода Bug Hunter { "findings": [ { "severity": "P0", "category": "security", "file": "db.py", "line": 12, "description": "SQL injection via f-string interpolation", "confidence": 0.95, "suggestion": "Replace with parameterized query" } ] }

5. Логировать и сравнивать

Сохраняйте результаты каждого раунда. Со временем вы увидите patterns: какие агенты какие баги находят лучше, какой порог оптимален, какие findings всегда false positive. Это данные для тюнинга.

⚠️ Типичные ошибки

❌ Запустить одинаковых агентов

Проблема: Запустили 3 копии одного промпта. Все нашли одни и те же баги. Параллелизм дал 0 benefit по качеству — только утроили стоимость.

Решение: Разные фокусы. Bug Hunter, Security Auditor, Code Quality. Разные промпты — минимум.

❌ Пропустить verify

Проблема: Запустили 3 агента, каждый нашёл по 10 находок. В итоге 30 «проблем», из которых 25 — ложные. Разработчик видит — и закрывает PR-ревью, потому что «это всё ерунда».

Решение: Adversarial verify — обязателен. Не optional.

❌ Не дать контекст

Проблема: Агентам дают только diff. Они не знают, что переменная query приходит из query string и уже проходит валидацию на уровне middleware — считают её «возможно опасной».

Решение: Давайте diff + маршруты + middleware + модели данных.

❌ Поверить каждому finding'у

Проблема: Агент сказал «SQL injection» — разработчик паникует. А на деле это строковый literal, который не подставляется извне. Без verify — ложная тревога.

Решение: Каждый finding проходит adversarial verify. Скептик проверяет эксплуатируемость и контекст.

⚔️ Сравнение со встроенным code-review

В Claude Code есть встроенный скилл code-review (вызывается через /code-review или как Skill tool). Он удобен для быстрой проверки перед коммитом, но у него есть ограничения.

Характеристика Встроенный /code-review Этот workflow
Количество агентов Один Три параллельных
Фокусы Generic code review Баги / Безопасность / Качество
Adversarial verify Нет Есть — скептик проверяет каждый finding
Настройка prompt'ов Нет Можно кастомизировать каждого агента
Параллельность Нет Через parallel() в Workflow tool
Когда использовать Быстрая проверка перед коммитом Глубокое ревью критичных изменений
💡 Когда что использовать: Встроенный /code-review — для быстрого «всё ли в порядке?» перед коммитом. Многоагентный workflow — для глубокого ревью критичных изменений (новый API, security-sensitive код, рефакторинг на сотни строк).

Итого: Многоагентный code review — это самодельный recipe из Workflow tool (parallel/pipeline) + Agent tool. Fan-out паттерн — общепринятый термин из параллельного программирования, не фича Claude Code. Три агента с разными фокусами находят больше, чем один. Adversarial verify фильтрует false positives. Не запускайте одинаковых агентов, не пропускайте верификацию и давайте контекст, а не только diff.
