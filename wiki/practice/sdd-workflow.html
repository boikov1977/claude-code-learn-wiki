📖 Что такое SDD и зачем это нужно

Открываешь Claude Code, говоришь «сделай тёмную тему» — агент кивает, пишет 200 строк, и ты уже не понимаешь, что там и зачем. Это vibe-coding: быстро, весело, и потом неприятно. Spec-Driven Development (SDD) — подход, где перед кодом создаётся спецификация: что меняется, почему, как устроено и из каких задач состоит реализация. Код — следствие спеки, а не наоборот.

Vibe-coding

  • Скажи Claude «сделай X» и надейся
  • Код появляется без плана
  • Через неделю непонятно, что и зачем
  • Изменение на 30 файлов за один промпт
  • Ревью — это мучение

Spec-Driven Development

  • Сначала описываешь что и зачем
  • Потом — как именно
  • Потом — конкретные задачи
  • Код — следствие трёх предыдущих шагов
  • Ревью — сверка с документом

Коротко: SDD — это когда ты сначала пишешь документ «что я буду делать и зачем», а потом Claude пишет код по этому документу. Не наоборот. Результат — осмысленный код с историей, которую можно найти в спеке, а не в куче git-коммитов.

📦 OpenSpec — npm-пакет и структура

@fission-ai/openspec — это npm-пакет, который даёт CLI и MCP-сервер для управления спецификациями. Он не просто хранит документы — он управляет жизненным циклом изменений: от proposal до архивации.

Установка

# Глобально npm install -g @fission-ai/openspec # Или в проекте npm install --save-dev @fission-ai/openspec

Структура проекта

OpenSpec создаёт директорию openspec/ с тремя ключевыми зонами:

Директория Назначение
specs/ Source of truth. Актуальные спецификации по доменам. Сюда вливаются дельта-спеки после синхронизации.
changes/ Активные изменения. Каждое — отдельная директория с proposal, design, tasks и дельта-спеками.
changes/archive/ Архив завершённых изменений. Формат: YYYY-MM-DD-name/.
💡 Файл .openspec.yaml в корне проекта определяет конфигурацию: какие артефакты нужны, какие артефакты обязательны перед реализацией, и какой workflow используется (например, spec-driven).

🔄 Полный цикл: Propose → Apply → Sync → Archive

Любой фича-рефакторинг-багфикс в OpenSpec проходит четыре этапа. Каждый этап — это не формальность, а осмысленный шаг:

1. Propose — определяем что и зачем

Суть: Создаём директорию изменения и генерируем артефакты: proposal.md, design.md, tasks.md и дельта-спеки.

Ключевые артефакты этапа:

Файл Отвечает на вопрос Содержимое
proposal.md Что и зачем Описание изменения, мотивация, ожидаемый результат. Для ревью — именно proposal читают первым.
design.md Как делаем Технический дизайн: какие файлы меняются, какие API затрагиваются, архитектурные решения.
tasks.md Какие конкретные шаги Чеклист задач в формате - [ ] Задача. Каждая задача — один файл или осмысленная группировка.
specs/ Что меняется в поведении системы Дельта-спеки: секции ADDED, MODIFIED, REMOVED, RENAMED. Описывают изменения требований.
# Пример: tasks.md ## Задачи - [ ] Создать CSS-переменные для тёмной темы - [ ] Добавить переключатель в header - [ ] Перевести компоненты на переменные - [ ] Сохранять выбор в localStorage - [ ] Добавить системную тему (prefers-color-scheme)

2. Apply — реализуем задачи

Суть: Читаем tasks.md и последовательно выполняем задачи. Каждая задача — мини-коммит. По завершении — отмечаем - [x].

Ключевые принципы этапа Apply:

  • Сначала читаем все контекстные файлы (proposal, design, specs, tasks)
  • Выполняем задачи последовательно, не пропускаем
  • Если задача неясна — останавливаемся и уточняем, не гадаем
  • Каждая задача — минимальное, сфокусированное изменение
  • По завершении задачи — сразу отмечаем в tasks.md

3. Sync — синхронизируем спеки

Суть: Дельта-спеки из changes/name/specs/ вливаются в основные specs/. Это не копирование — это интеллектуальное слияние.

Формат дельта-спек:

## ADDED Requirements ### Requirement: Dark Theme Support Система SHALL предоставлять переключатель тёмной темы. ## MODIFIED Requirements ### Requirement: Header Navigation #### Scenario: Add theme toggle - WHEN пользователь нажимает на иконку темы - THEN тема переключается и сохраняется ## REMOVED Requirements ### Requirement: Legacy Color System

При синхронизации ADDED требования добавляются в основную спеку, MODIFIED — интеллектуально мержатся (не заменяют целиком), REMOVED — удаляют соответствующие блоки.

4. Archive — архивируем завершённое

Суть: Изменение перемещается в archive/YYYY-MM-DD-name/. Все артефакты сохраняются — потом можно вернуться и посмотреть, что делали.

Перед архивацией OpenSpec проверяет:

  • Все ли артефакты в статусе done
  • Все ли задачи в tasks.md отмечены как - [x]
  • Нужно ли синхронизировать дельта-спеки (предлагает опцию)
💡 Архив — это не мусорка. Это журнал истории проекта. Через полгода ты можешь открыть архив и точно вспомнить, что делал, почему, и какие решения принял. Попробуй сделать это с vibes-coding.

🔥 Пример: добавляем тёмную тему

Полный walkthrough — от идеи до архивации. Чтобы было наглядно.

Шаг 1: Создаём изменение

$ openspec new change "add-dark-theme" Created: openspec/changes/add-dark-theme/ Schema: spec-driven

Шаг 2: Пишем proposal.md

# Proposal: Add Dark Theme ## Problem Пользователи работают ночью и в тёмных помещениях. Белый интерфейс слепит глаза и вызывает усталость. ## Solution Добавить переключатель тёмной темы с сохранением выбора в localStorage и поддержкой системных настроек. ## Success Criteria - Переключатель виден в header - Тема сохраняется между сессиями - Автоматически определяется системная тема - Все компоненты корректно отображаются в обеих темах

Шаг 3: Пишем design.md

# Design: Dark Theme ## Approach CSS-переменные (custom properties) на корневом элементе. Два набора: :root (светлая) и [data-theme="dark"] (тёмная). ## Files to Modify - src/styles/variables.css — добавить тёмные переменные - src/components/Header.tsx — добавить переключатель - src/hooks/useTheme.ts — новый хук для управления темой - src/styles/components.css — перевести на переменные ## Architecture Decisions - Хранение: localStorage + data-атрибут на <html> - Фоллбэк: prefers-color-scheme media query - Переключение: без перезагрузки страницы

Шаг 4: Пишем tasks.md

## Tasks - [x] Создать variables.css с набором CSS-переменных для обеих тем - [x] Добавить хук useTheme с логикой переключения и localStorage - [x] Добавить кнопку-переключатель в Header.tsx - [x] Перевести components.css на CSS-переменные - [x] Добавить media query для системной темы - [x] Проверить все компоненты в обеих темах

Шаг 5: Пишем дельта-спеку

# openspec/changes/add-dark-theme/specs/ui/spec.md ## ADDED Requirements ### Requirement: Dark Theme Support Система SHALL предоставлять визуальный переключатель между светлой и тёмной темами. #### Scenario: User toggles theme - WHEN пользователь нажимает кнопку переключения темы - THEN тема меняется мгновенно без перезагрузки - AND выбор сохраняется в localStorage #### Scenario: System theme detection - WHEN пользователь открывает страницу впервые - THEN система определяет предпочтение через prefers-color-scheme ## MODIFIED Requirements ### Requirement: Header Navigation #### Scenario: Theme toggle in header - WHEN пользователь смотрит на header - THEN он видит иконку переключения темы справа от навигации

Шаг 6: Реализуем задачи

Claude (или ты) читает tasks.md и последовательно реализует каждую задачу. По завершении каждой — отмечает - [x] в tasks.md. Код пишется по design.md, а не «на ощупь».

Шаг 7: Синхронизируем спеки

$ openspec sync Synced delta specs for add-dark-theme: ui: Added "Dark Theme Support", Modified "Header Navigation" Main specs updated.

Шаг 8: Архивируем

$ openspec archive Archived: add-dark-theme Location: openspec/changes/archive/2026-06-17-add-dark-theme/ Specs: synced Tasks: 6/6 complete

Итого: От идеи до архивации — 8 шагов. В каждом я точно знаю, что делаю и зачем. Никакого vibe-coding.

✅ Best practices

Пиши spec перед кодом

Самый главный принцип. Если ты не можешь описать изменение в proposal и tasks — ты не понимаешь, что делаешь. А если не понимаешь — зачем писать код? Спека — это способ заставить себя подумать перед действием.

Дроби на мелкие изменения

Одно изменение = одна осмысленная фича или один рефакторинг. Не «переделать весь фронт», а «добавить тёмную тему». Если tasks.md больше 10 задач — возможно, стоит разбить на два изменения.

Обновляй spec если передумал

Реализация показала, что дизайн неправильный? Не пиши код «как получится» — обнови design.md и tasks.md. OpenSpec поддерживает fluid workflow: можно вернуться к редактированию артефактов на любом этапе.

Читай proposal перед review

Когда делаешь code review или вернулся к проекту через месяц — начни с proposal.md. Там написано зачем всё это делалось. Git-коммиты говорят тебе что изменилось, proposal — почему.

Используй дельта-спеки для коммуникации

Дельта-спеки — это язык общения между разработчиком и системой. Формат ADDED/MODIFIED/REMOVED читается и человеком, и Claude. Это не бюрократия — это точность.

⚠️ Типичные ошибки

«Напишу spec потом, а то теряю поток»

Потом = никогда. А потом выясняется, что ты сделал не то, потому что не подумал. И тратишь 2 часа на переделку вместо 20 минут на spec. Поток — это иллюзия. Spec — это реальность.

«Одно изменение на 30 файлов»

Если tasks.md похож на роман, ты сделал слишком много. Разбей на отдельные изменения. Каждое — одна осмысленная единица. Архивировать роман — мучение. Архивировать абзац — pleasure.

«Сделал по design, а он оказался неправильным»

Не молчи и не коди «как получится». Обнови design.md и tasks.md. Скажи Claude: «Обнови артефакты, вот что изменилось». Это не провал — это процесс. Spec живой, не каменный.

«Синхронизирую спеки после каждого коммита»

Не надо. Синхронизация — это один раз после завершения всех задач. Дельта-спеки живут в changes/name/specs/ и спокойно ждут своего часа. Не торопись сливать до того, как всё заработает.

«Archive — это лишнее, я и так помню»

Через месяц — помнишь. Через полгода — нет. Архив не для тебя сегодня. Архив для тебя через полгода. И для тиммейта, который пришёл и спрашивает «зачем мы это сделали».

Итого: SDD через OpenSpec — это не бюрократия, а инструмент ясного мышления. Proposal (что и зачем) → Design (как) → Tasks (шаги) → Apply (код) → Sync (спеки) → Archive (история). Каждый этап даёт ответ на конкретный вопрос. Результат — код с документацией, документация с историей, история с мотивацией. А не кучка коммитов «fix bug».
