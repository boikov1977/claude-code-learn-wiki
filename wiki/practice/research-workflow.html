Deep Research Workflow

Самодельный рецепт автоматизированного исследования — поиск, верификация и синтез отчёта через MCP-инструменты

Это не встроенная фича Claude Code. Deep Research Workflow — это самодельный рецепт (workflow), собранный из доступных инструментов: MCP-серверов, Workflow tool и методик оркестрации. В Claude Code есть встроенный скилл deep-research, но он работает иначе — см. сравнение в конце статьи.

Этот workflow не поставляется «из коробки». Его собирают вручную: подключают MCP-серверы с инструментами для веб-поиска и чтения страниц, настраивают Workflow tool для оркестрации, и прописывают промпты для каждой фазы. Преимущество — полный контроль над каждым шагом и возможность подставить любые источники.

🔍 Зачем это нужно

Допустим, тебе нужно понять, какие MCP-серверы сейчас популярны, или собрать обзор инструментов для конкретной задачи. Вручную — это десятки вкладок, копирование, сравнение, сомнения в актуальности. Claude Code умеет делать это автоматически: разбить вопрос на подуглы, параллельно пробежаться по источникам, проверить факты перекрёстно и выдать готовый отчёт со ссылками.

Суть: Deep Research Workflow — это не «погуглить». Это систематическое исследование: многократный поиск, чтение нескольких источников, проверка противоречий и синтез в структурированный отчёт. Результат — не «гугл-выдача», а документ, на который можно опереться. Но это твой рецепт — не готовая функция платформы.

Сравни два подхода:

❌ Ручной поиск

  • Открыл Google, ввёл запрос
  • Пролистал 10 ссылок, открыл 3 в новых вкладках
  • Прочитал, забыл, что было в первой
  • Не уверен, актуальны ли данные
  • Нет ссылок в итоговом тексте
  • 30–60 минут рутины

✅ Deep Research Workflow

  • Claude разбивает запрос на 5 углов
  • Параллельно ищет по каждому
  • Читает топ-15 источников
  • Перекрёстно проверяет факты
  • Генерирует отчёт со ссылками и confidence
  • 5–10 минут без твоего участия

🛠️ Инструменты

Всё строится на MCP-серверах — внешних процессах, которые Claude подключает через stdio и вызывает как обычные инструменты. Для этого workflow ключевой — poisk-mcp, но подойдёт любой MCP-сервер с инструментами поиска и чтения.

poisk-mcp: инструменты для исследования

Инструмент Назначение Когда использовать
search_web Поиск по запросу. Возвращает список ссылок с описаниями и оценками релевантности. Первый шаг: собрать потенциальные источники. Запускай с разных формулировок.
read_webpage Чтение содержимого страницы. Конвертирует HTML в чистый текст. Когда нашёл перспективную ссылку — прочитай её полностью.
browse_summarize Чтение + автоматическое извлечение ключевых тезисов. Для быстрого сканирования: «этот источник релевантен или нет?»
web_research Глубокое исследование: поиск + чтение топ-страниц + анализ + отчёт. Всё в одном вызове. Когда нужен готовый отчёт, а не сырые ссылки. depth=1–5.

Примеры вызовов

# Широкий поиск — собрать источники search_web("MCP servers popular 2026 Claude Code", max_results=10) # Конкретный вопрос search_web("best MCP servers for code review", max_results=8) # Прочитать конкретную статью read_webpage("https://example.com/mcp-overview") # Быстрое резюме длинной статьи browse_summarize("https://example.com/mcp-guide", "Какие MCP-серверы описаны?") # Глубокое исследование за один вызов web_research("MCP экосистема 2026: популярные серверы и тренды", depth=4)

Workflow tool — оркестрация

Сам workflow orchestrates через Workflow tool: веерный (fan-out) поиск, параллельное чтение, последовательная верификация и синтез. Это не один вызов инструмента — это цепочка шагов, каждый из которых управляет предыдущими.

Альтернативы: другие MCP-серверы

Если poisk-mcp недоступен, подойдут другие MCP-серверы с инструментами для работы с вебом: Context7, Tavily, Firecrawl и т.д. Ключевое требование — наличие хотя бы двух инструментов: поиск (найти ссылки) и чтение (получить содержимое страницы). Встроенный WebSearch Claude Code через OpenCode-прокси не работает (ошибка схемы DeepSeek), поэтому MCP обязателен.

1

Scout — веерный поиск по теме

Один запрос в Google даёт одну перспективу. Хорошее исследование — это 5 параллельных поисков с разных ракурсов. Claude сам разбивает исходный вопрос на подуглы и запускает search_web по каждому.

Пример: Вопрос «Какие MCP-серверы популярны?» разбивается на:

  1. "MCP servers popular 2026 GitHub stars" — количественные метрики
  2. "best MCP servers Claude Code production" — продакшн-опыт
  3. "MCP ecosystem overview Anthropic" — официальная позиция
  4. "MCP servers comparison review" — сравнительные обзоры
  5. "MCP new servers trending" — свежие новинки

Каждый поиск возвращает 8–10 результатов. После дедупликации по URL получается ~30–40 уникальных ссылок.

2

Deep-read — параллельное чтение лучших источников

Из ~35 ссылок отбираются 15 самых перспективных (по релевантности и разнообразию доменов). Каждая читается через read_webpage или browse_summarize. Параллельно — не нужно ждать, пока прочитаешь первую, чтобы начать вторую.

Что искать в каждом источнике

  • Конкретные имена и цифры — названия серверов, звёзды, даты
  • Цитаты и примеры — прямые выдержки, которые можно проверить
  • Дата публикации — устаревшая информация = низкий приоритет
  • Автор и площадка — Anthropic blog vs личный пост в dev.to

Из каждой страницы извлекаются фальсифицируемые утверждения — конкретные факты, которые можно подтвердить или опровергнуть другим источником. Например: «Context7 — самый популярный MCP-сервер с 15K+ звёзд» — это фальсифицируемое утверждение.

3

Verify — проверка фактов через несколько независимых источников

Это самая важная фаза. Каждый факт, собранный из источников, проверяется минимум двумя независимыми источниками. Если факт подтверждается только одним — он помечается как low confidence.

Верификация проверяет не только факты, но и противоречия: если один источник говорит «MCP-сервер X deprecated», а другой «активно развивается» — это нужно зафиксировать и указать в отчёте.

4

Synthesize — отчёт с confidence-оценками

Последний шаг: все проверенные факты, противоречия и источники объединяются в структурированный отчёт. Не «список ссылок», а аргументированный текст с ответом на исходный вопрос.

Структура отчёта

  • Executive summary — ответ на вопрос в 2–3 предложениях
  • Детали — развёрнутый ответ с аргументами
  • Источники — список ссылок, по которым собрана информация
  • Confidence — насколько можно доверять каждому тезису
  • Оговорки — что не удалось проверить или где есть противоречия

📋 Пример: популярные MCP-серверы в Claude Code

Давай пройдём конкретный пример — как этот workflow ответит на вопрос: «Какие MCP-серверы сейчас популярны в Claude Code?»

Фаза 1: Scout

# Claude запускает 5 параллельных поисков: search_web("most popular MCP servers GitHub stars 2026", max_results=10) search_web("best MCP servers Claude Code production setup", max_results=10) search_web("MCP ecosystem Anthropic official servers", max_results=8) search_web("MCP servers comparison review 2026", max_results=8) search_web("new MCP servers trending Claude", max_results=8) # Результат: ~38 ссылок, после дедупликации — 29 уникальных

Фаза 2: Deep-read

# Claude выбирает 12 самых перспективных и читает: read_webpage("https://github.com/topics/model-context-protocol") # → список самых популярных репозиториев по звёздам browse_summarize("https://docs.anthropic.com/en/docs/claude-code/mcp", "Какие MCP-серверы упоминает Anthropic?") # → Context7, Brave Search, filesystem, memory, postgres read_webpage("https://habr.com/ru/articles/mcp-servers-overview") # → сравнительная таблица серверов read_webpage("https://github.com/modelcontextprotocol/servers") # → официальный репозиторий MCP servers # ... и ещё 8 страниц

Фаза 3: Verify

Пример проверки:

Утверждение: «Context7 — самый популярный MCP-сервер»

  • GitHub Topics — Context7 на первом месте по звёздам → подтверждает
  • Anthropic docs — Context7 упоминается первым в рекомендациях → подтверждает
  • Habr-статья — в таблице Context7领先 → подтверждает

Пример противоречия:

Утверждение: «Brave Search MCP deprecated»

  • Reddit-пост — говорит, что deprecated → подтверждает
  • GitHub repo — последний коммит 2 недели назад, активные issues → опровергает
  • Anthropic docs — Brave Search всё ещё в списке рекомендованных → опровергает
  • Confidence: LOW — 2 из 3 источников опровергают, но факт включается в оговорки

⭐ Best practices

Не полагайся на один источник

Минимум 3 независимых источника на каждый ключевой тезис. Один источник — это не исследование, а пересказ. Чередуй: официальная документация, GitHub, Habr/dev.to, Reddit, личные блоги.

Ищи противоречия

Если все источники говорят одно и то же — это подозрительно. Возможно, все они скопировали информацию из одного места. Специально ищи альтернативные точки зрения и включай их в отчёт.

Указывай confidence

Каждый тезис в отчёте должен иметь уровень уверенности: HIGH / MEDIUM / LOW. Это помогает читателю понять, на что можно полагаться, а что требует дополнительной проверки.

Повторяй verify при low-confidence

Если много тезисов с LOW confidence — запусти ещё раунд верификации с другими поисковыми запросами. Часто проблема не в источниках, а в формулировке поискового запроса.

Формула хорошего отчёта

# Хороший отчёт = формула Ответ на вопрос (2-3 предложения) + Контекст (почему это важно, что изменилось) + Аргументы (факты с источниками) + Противоречия (если есть — честно указать) + Confidence (уровень уверенности по каждому тезису) + Источники (список ссылок с датами) + Оговорки (что не удалось проверить)

⚠️ Типичные ошибки

Принимать первый результат за истину

Ошибка: Прочитал одну статью и выдал её как отчёт.

Почему плохо: Статья может быть устаревшей, неточной, рекламной или просто ошибочной. Один источник — это не исследование, а пересказ.

Решение: Минимум 3 независимых источника на каждый ключевой тезис.

Не проверять авторитетность источника

Ошибка: Личный твит и официальная документация взвешиваются одинаково.

Почему плохо: Anthropic blog > сравнительный обзор на Habr > мнение в комментариях. Источники имеют разный вес.

Решение: При верификации учитывай кто пишет и где.

Игнорирование дат

Ошибка: Ссылаешься на статью 2024 года как на актуальный источник.

Почему плохо: MCP-экосистема развивается быстро. Инструмент, который был popular в 2024, мог стать deprecated к 2026. Устаревшая информация хуже, чем отсутствие информации.

Решение: Всегда проверяй дату публикации. Приоритизируй источники не старше 6 месяцев.

Все ссылки с одного домена

Ошибка: 5 из 5 источников — habr.com (или dev.to, или medium).

Почему плохо: Одна площадка = один угол зрения. Habr склонен к техническим подробностям, Medium — к обзорам, Reddit — к мнениям. Без разнообразия ты видишь только одну грань.

Решение: MMR-разнообразие (встроено в poisk-mcp) помогает, но и сам Claude должен следить за балансом доменов.

Чек-лист перед выдачей отчёта

  • ☑ Минимум 3 источника на каждый ключевой тезис
  • ☑ Все факты имеют дату публикации источника
  • ☑ Факты отделены от мнений
  • ☑ Противоречия честно описаны
  • ☑ Confidence указан для каждого тезиса
  • ☑ Источники с разных доменов
  • ☑ Оговорки про непроверенные данные

🔍 Сравнение со встроенным deep-research

В Claude Code есть встроенный скилл deep-research (вызывается через Skill tool или /deep-research). Он тоже ищет в вебе, но работает иначе — и это принципиальное отличие.

Характеристика Встроенный deep-research Этот workflow
Оркестрация Встроенная, не настраивается Кастомная через Workflow tool
Инструменты Внутренние (WebSearch, WebFetch) Любые MCP-серверы (poisk-mcp, Context7 и др.)
Parallel / pipeline Не применяется Веерный поиск, параллельное чтение
Верификация фактов Нет Adversarial verify с confidence-оценками
Контроль Чёрный ящик Полный контроль над каждым шагом
